## 1. 调研目标与决策变化

本文闭环以下问题：

1. React Native 单客户端能否承载竖屏轻量分层 2D 办公室；
2. 普通 RN 组件在 4 房间、8 员工和 3 并行动画下是否达标；
3. 何时允许把角色与任务道具绘制层迁移到 Skia `Atlas`；
4. 静态分层资源、有限姿态、路径、遮挡、标签和无障碍如何协作；
5. 办公室如何与任务、原生结果页和快照恢复形成交付闭环；
6. 黄金视觉如何在批量生产前控制主观风格、授权与一致性风险；
7. 三类 AI 任务的质量、延迟、成本和供应商替换；
8. IAP、成本预占、双账本、Temporal、PostgreSQL、文件、安全、HR、经济和现实时间是否继续成立。

**决策变化：此前的双运行时客户端结论已被新的产品约束和交互范围推翻。** 当前及备用架构均不再保留第二客户端运行时。办公室被明确约束为固定房间、分层静态资源、有限姿态、语义槽位、受控路径、最多 8 人和最多 3 个并行动画，因此单一 React Native 树是更直接、可测试、可访问且生命周期风险更低的基线。

“工程信心 >95%”表示已经有明确方案、官方能力边界、可量化门槛和可逆的条件式优化，不表示主观视觉或未经真机测试的性能必然成功。主观视觉只通过黄金视觉签字控制；性能只通过 Release 真机数据确认。

### 1.1 闭环结果

| 事项 | 最终结论 | 工程信心 | 剩余证据 |
|---|---|---:|---|
| 客户端路线 | React Native 0.86.x 单客户端 | >95% | 双平台 Release PoC |
| 默认场景 | View / Image / Pressable 分层渲染 | >95% | 20 分钟压力 Gate |
| 动画与手势 | Reanimated 4 + Gesture Handler，精确补丁 PoC 后冻结 | >95% | RN 0.86 双平台兼容矩阵 |
| 条件式优化 | 仅压力失败后迁移 actors-and-props 到 Skia Atlas | 95% | 同夹具前后对比 |
| 局部复杂动作 | Rive 或 Sprite Atlas，必须有静态替代 | 94% | 动作 PoC 与资源预算 |
| 场景数据 | OfficeSlot / OfficePath / EmployeeSceneState / SceneSnapshot / visual_intent | >95% | 快照乱序与恢复测试 |
| 结果呈现 | RN 场景汇报演出 + 原生 WorkProduct 页面 | >95% | 用户理解与无障碍测试 |
| 黄金视觉 | 单办公室小批量签字后再量产 | 95% | 商业授权、风格与双平台签字 |
| AI / IAP / 成本 | 平台托管多供应商 + IAP + 原子预占 + 双账本 | 93% | 真实任务、合同和账单对账 |
| Temporal / PostgreSQL | 耐久流程 + 权威事实与追加账本 | 95% | Replay、并发和恢复测试 |
| 文件与隐私 | 隔离解析、租户隔离、来源级删除和区域锁定 | 92% | 首发地区与合同条款 |
| HR / 经济 / 时间 | 服务端状态机、幂等结算、UTC + IANA 时区 | 95% | 属性测试与内测数值 |

上述比例是技术准备度，不是统计概率。

## 2. 单客户端渲染决策

### 2.1 产品硬边界

办公室必须支持：

- 一个竖屏楼层、3—4 个房间和日 / 夜状态；
- 全景重点 4—6 人，当前楼层最多 8 人；
- 同时移动最多 3 人；
- 每名员工一个权威槽位、一个主要动作和一个 `primary_visual_task`；
- 分层静态背景、家具、前景遮挡、透明员工和任务道具；
- 有限姿态、房间聚焦、受限缩放和平移；
- 原生标签、气泡、卡片、Sheet、文件、长文、支付和无障碍；
- 场景失败或资源缺失时业务页面仍完整可用。

明确不需要自由旋转、真实物理、任意家具摆放、逐帧服务端坐标或场景内复杂排版。该范围决定了问题本质是“原生 UI 中的分层互动场景”，而不是独立实时模拟。

### 2.2 候选实现比较

| 方案 | 生命周期 | 原生 UI / 无障碍 | 8 人绘制能力 | 复杂度 | 结论 |
|---|---|---|---|---|---|
| 普通 RN View / Image / Pressable | 与 App 一致 | 最佳 | 需压力验证 | 最低 | 默认基线 |
| RN + Skia Atlas 局部绘制 | 与 App 一致 | Overlay 继续原生 | 批量精灵更高效 | 中 | 条件式优化 |
| RN + Rive 局部动作 | 与 App 一致 | Overlay 继续原生 | 适合少量状态机动作 | 中 | 局部可选 |
| 动画 WebP 全量承载 | 解码路径有平台差异 | 点击与语义需额外覆盖 | 资源与内存难统一 | 中 | 不作唯一基线 |
| WebView + Three.js | 多一套渲染与事件边界 | 无障碍和平台集成较弱 | 需额外 GPU / 生命周期验证 | 高 | 不采用 |
| iOS / Android 双套场景 | 平台直接 | 可做 | 逻辑和资源重复 | 很高 | 不采用 |

普通 RN 的优势不是“永远性能最佳”，而是当前规模下能以最小架构复杂度共享导航、状态、无障碍、生命周期、错误边界和测试工具。Skia `Atlas` 的价值是当大量精灵成为已测瓶颈时替换单层绘制，而不是建立第二产品路线。

### 2.3 最终决策

1. 客户端固定为 React Native 0.86.x + TypeScript + New Architecture + Hermes；
2. 场景默认使用 `View`、`Image`、`Pressable`；
3. 位移、缩放、淡入和房间聚焦使用与 RN 0.86 兼容的 Reanimated 4；
4. 双指缩放、受限平移、点击竞争使用 Gesture Handler；
5. 两项动画 / 手势依赖的具体补丁在双平台 PoC 后精确冻结，不在调研文档中猜测版本；
6. 背景使用经双平台验证的 JPEG、PNG 或静态 WebP；透明层优先 PNG，静态 WebP 需验证；
7. 复杂动作可局部使用 Rive 或 Sprite Atlas，并提供静态姿态；
8. 普通 RN 未通过压力 Gate 且完成两轮优化后，才迁移 `actors-and-props` 到 Skia `Atlas`；
9. 迁移不改变数据、交互、人数、业务能力和原生 Overlay；
10. 不使用 WebView 或 Three.js，不引入第二场景运行时。

### 2.4 条件式优化触发器

只有同时满足以下条件才允许 Skia 迁移：

- 使用 Release 真机而非模拟器或开发包；
- 夹具固定为 4 房间、8 员工、3 并行动画、1 气泡、2 展开标签；
- 已排除无关 React 重渲染、图片解码峰值、JS 回调过多、标签布局抖动和后台任务；
- 已完成两轮定向优化并保存前后数据；
- P95 UI 帧时间仍 >20 ms，或 >100 ms 非加载帧仍 >1 次 / 分钟，或场景增量内存明显无法接近 120 MB 目标；
- 评审确认瓶颈来自角色与道具绘制，而不是业务列表、网络、日志或图片尺寸。

迁移后必须用完全相同夹具复验。如果仍失败，继续减少绘制开销或修复资源，而不是拆出另一客户端，也不以降低核心业务、隐藏员工或取消恢复测试来宣布通过。

## 3. 场景数据与一致性闭环

### 3.1 语义数据优于逐帧坐标

服务端负责“谁在什么语义位置、真实任务处于什么阶段、允许表达什么”，客户端负责“该布局版本下如何显示”。核心对象：

- `OfficeSlot`：room、slot type、normalized x/y、scale、zIndex、facing、labelAnchor、supportedPoses；
- `OfficePath`：起点、终点、路径节点、节点 zIndex 和 facing；
- `EmployeeSceneState`：employee、authoritativeSlot、primaryAction、primaryVisualTask、pose 和可选 path；
- `SceneSnapshot`：sceneVersion、serverSequence、layoutVersion、companyTime、roomStates、employees、visualIntents；
- `visual_intent`：事实来源、员工、任务、意图类型、目标槽位、优先级、中断策略、有效期和安全进度语义。

服务端不发送逐帧 x/y。客户端按 `layout_version` 映射槽位和路径；未知版本或缺失节点回退到安全槽位和静态姿态，并请求快照。这降低网络频率、避免设备差异导致服务端坐标失真，也让布局资源可独立发布和回滚。

### 3.2 分层与遮挡

固定层级为：

```text
background
floor
rear-furniture
actors-and-props
front-mask
native-overlays
```

所有可绘制元素显式声明 `zIndex`。员工经过门框、桌前或走廊时，`OfficePath` 节点决定当前层级；前景遮挡由静态 `front-mask` 完成。原生文字、标签、气泡、热区和 Sheet 始终位于 `native-overlays`，不烘焙进图像。这样既保留空间层次，也避免自由深度推断和不可测试的遮挡规则。

### 3.3 并发与恢复

- 一名员工只有一个权威槽位；
- 一名员工只播放一个主要动作；
- 一名员工只有一个 `primary_visual_task`；
- 同时最多 3 个员工移动，多余意图排队或原地表达；
- 本地动画完成只清理视觉队列，不能推进任务；
- `output.ready` 之前不能显示完成或走向汇报槽位；
- 断线、前后台或进程恢复后，以更高 `sceneVersion` 快照替换本地事实；
- 旧 `serverSequence` 事件不得覆盖新快照；
- 快照不包含完整回答、用户文件、访问令牌或供应商密钥。

## 4. 移动性能与生命周期闭环

### 4.1 正式验收夹具

Release 真机运行 20 分钟：4 房间、8 员工、3 并行动画、1 气泡、2 展开标签；期间执行全景 / 聚焦、员工点击、结果页往返、网络中断恢复和前后台切换。场景默认使用普通 RN 组件。

| 指标 | 门槛 |
|---|---|
| 流畅度 | 接近稳定 60 FPS |
| UI 帧时间 | P95 ≤20 ms |
| 严重卡顿 | >100 ms 非加载帧 ≤1 / 分钟 |
| 场景增量内存 | 相对无办公室基线目标 ≤120 MB |
| 生命周期 | 50 次前后台无状态错乱 |
| 快照恢复 | 丢失本地动作后恢复服务端当前状态 |
| 交互 | 常规点击反馈 P95 ≤200 ms |
| 结果 | `output.ready` 后跳过演出立即可见 |

“接近稳定 60 FPS”要求同时查看 P95 帧时间和长帧，不能只报告平均 FPS。120 MB 是目标而非允许忽略平台差异的绝对承诺；iOS 与 Android 分开记录，并保留图片解码峰值与稳态快照。

### 4.2 优化顺序

1. selector、memo、稳定 key 和组件边界，消除无关重渲染；
2. 连续动画留在 Reanimated 工作线程；
3. 避免每帧 setState、JS 计时器和大对象分配；
4. 按显示尺寸导出与解码图片，限制透明大图同时驻留；
5. 合并不变背景层，减少阴影、模糊和布局测量；
6. 不可见员工停止循环，保持 8 人 / 3 移动硬上限；
7. 检查标签、气泡与无障碍节点的布局成本；
8. 两轮仍失败才启用 Skia Atlas 绘制层。

### 4.3 生命周期

办公室是普通 RN 路由，不存在额外运行时加载、暂停或卸载。App 后台停止动画；云端 AI、文件、工资和定时流程继续。返回前台先读取增量或快照再恢复。结果页打开时办公室路由失焦或卸载，返回后使用新快照；不补播全部离线过程。该模型显著减少黑屏、音频占用和多运行时状态分叉风险，但仍必须做 50 次前后台与进程被杀恢复测试。

## 5. 黄金视觉与资源产能

### 5.1 黄金视觉 Gate

批量生产前只制作：

- 1 个竖屏办公室；
- 3—4 个房间；
- 4 名正式员工；
- 6 个核心姿态：idle、walk、work、read、report、blocked；
- 日 / 夜状态；
- 全景与房间聚焦；
- 原生员工标签、状态气泡和点击热区。

签字项包括：目标 iPhone / Android 构图、房间辨识、员工辨识、姿态语义、前景遮挡、文字对比、动态字体、减少动态效果、资源解码与内存、商业授权、来源记录和风格一致性。产品与视觉负责人书面确认后才允许批量生产。该 Gate 控制主观风险，但不声称所有用户都会喜欢。

### 5.2 资源格式结论

- 背景：经双平台验证的 JPEG、PNG 或静态 WebP；
- 透明角色 / 道具 / 遮挡：优先 PNG，静态 WebP 需验证 alpha、色彩和内存；
- 动画 WebP：不作为唯一基线；
- 复杂局部动作：Rive 或 Sprite Atlas，必须可中断且有静态替代；
- 文字、标签、气泡、卡片、Sheet：原生 RN UI；
- 源资产：位图或矢量，记录导出尺寸、透明边界、锚点、许可和压缩规则。

### 5.3 scene manifest 与自动校验

manifest 记录 layoutVersion、设计画布、层级、rooms、slots、paths、occlusionZones、dayNightVariants、assets 和 licenseManifestVersion。每个资源记录 ID、源引用、格式、尺寸、倍率、色彩、透明边界、平台、许可、作者、来源和哈希。每个姿态记录脚底与标签锚点、facing 和包围盒。

CI 阻断：错误尺寸 / 倍率、格式不在白名单、透明边界异常、锚点越界、槽位超出 0—1、路径端点不存在、zIndex 缺失、遮挡引用失效、资源未引用、ID 重复、许可缺失 / 过期、压缩超预算、哈希不符或 Schema 不兼容。远端 manifest 必须签名、版本化、可回滚，并保留上一稳定版本。

## 6. AI 质量、成本、IAP 与人格闭环

### 6.1 多供应商中立层

模型质量、价格、限额、数据政策和区域会变化。产品继续采用“App Store IAP 自动续期订阅 + 平台托管多供应商 AI”：用户不提供 API Key；平台采购供应商能力，以订阅净收入承担模型成本，通过订阅周期算力额度控制单用户支出。

AI Gateway 的中立协议包含 capability_profile、input_references、required_output_schema、tool_policy、safety_context、data_region、retention_class、latency_budget、usage_budget、entitlement_cycle_id、reservation_id 和 model_price_version_id。供应商适配层处理端点、流式、工具、结构化输出、用量与错误差异。自动切换只允许进入已通过质量评测、价格有效、健康正常、区域兼容且用户明确同意的路线；没有候选时可恢复阻塞。

### 6.2 模型分层与正式成果

| 层级 | 职责 | 约束 |
|---|---|---|
| 快速结构层 | 意图、缺失信息、分类、秘书摘要 | Schema 输出，不做最终事实结论 |
| 核心交付层 | 写作、调研、文件 / 数据分析 | 已验证路线，强制来源与结构 |
| 验证层 | 要求覆盖、引用、计算、安全 | 优先确定性代码，模型不代替规则 |
| 人格表达层 | 员工短汇报和进度 | 只读 canonical output，不新增事实 |

`WorkProduct` 包含 answer_blocks、claims、numbers、citations、limitations、unresolved_questions 和 artifact_refs。`PersonaReport` 只能引用已有 claim_id，数字、来源、限制与结论不得出现新集合；失败时回退标准模板。

三类任务各不少于 150 条，总计不少于 450 条；至少 20% 对抗 / 边界、20% 含文件或表格、20% 多轮修订。发布门槛：要求覆盖双评 ≥4.2/5；来源支持率 ≥95%；严重错配 <1%；关键字段准确率 ≥98%；人格严重事实漂移 =0；工具越权 =0；失败可恢复或清晰降级 ≥99%。

### 6.3 成本预占与双账本

每个订阅产品定义周期 AI 算力额度。模型成本目标为 P75 用户不超过订阅净收入 25%，设计硬上限 30%，最终价格与额度以真实任务和 App Store 结算验证。

每次调用先按 `ModelPriceVersion`、预计输入 / 输出、缓存与工具计算最大费用，在 PostgreSQL 同一事务原子创建 UsageReservation。调用后按供应商用量写 ProviderUsageLedgerEntry，并按交付结果写 UserAllowanceLedgerEntry；释放差额，重试不能重复扣费。设置用户周期、单任务、供应商日 / 月和平台全局硬上限。每日对账，差异 >3% 告警并暂停扩大流量。

### 6.4 IAP 与平台密钥

客户端使用 StoreKit 2 和 `appAccountToken`；服务端通过 App Store Server Notifications V2 与 Server API 维护订阅、退款、撤销、宽限期、过期和恢复购买。首版不提供外部付款入口、网页购买引导或 BYOK。

平台供应商 Key 存入 KMS / Secret Manager，按环境、供应商和区域隔离；只有 AI Worker 最小权限身份可临时取得。Key 不进入客户端、Temporal History、普通数据库字段、日志、分析、客服后台或崩溃报告。运营后台可测试、轮换、停用和熔断，但不能显示完整 Key；所有操作审计。

### 6.5 当前优先 Adapter PoC

当前验证可先接 DeepSeek V4 文本能力，但不改变生产多供应商架构。使用官方 API 域名和后台允许清单，不依赖已公告下线的旧别名。契约测试覆盖普通响应、SSE、空事件、取消、JSON、Tool Calls、用量、401、402、429、500 和 503。文件、图片与表格先由受控 OCR / 解析器转为带来源文本。PoC 使用合成或非敏感数据；目标地区的条款、隐私、处理区域和合同在生产 Gate 复核。

## 7. 结果导航与原生页面闭环

最终交付顺序：

```text
output.ready
→ RN 场景员工整理并走向汇报槽位
→ 用户触发 openResult(task_id, work_product_version)
→ 原生员工汇报页查询权威 WorkProduct
→ 展示 PersonaReport + 正文 + 来源 / 限制 / 文件 / 操作
→ 返回办公室后重新应用 SceneSnapshot
```

办公室负责“谁完成了、正在做什么、何时汇报”的空间叙事；原生页面负责长文、表格、代码、文件预览、选择、复制、分享、导出、支付和无障碍。`output.ready` 前不能提前汇报；结果可用后跳过走路立即打开。多个结果由秘书汇报收件箱聚合。场景资源失败时同一收件箱仍能打开正式结果。

未知持续时间的模型任务显示持续活动和已发生的真实阶段，不显示虚构百分比。原生 `openResult` 是路由意图，不是业务状态变更。该交付方案的工程信心 >95%；剩余验证是用户是否理解“员工已完成—点击查看正式报告”，以及屏幕阅读、动态字体和返回快照是否顺畅。

## 8. 虚拟经济与防刷闭环

经济系统继续以 `root_task_id` 为主结算单元：首次成功交付产生一次收入；追问、补充、返回修改和重试不重复发基础收入；执行前取消或无有效交付不产生收入；系统恢复后仍以同一 `settlement_unit_key` 结算。虚拟收入不影响订阅、AI 额度，也不可兑换真实货币。

```text
gross_revenue = base_value(task_value_class)
eligible_revenue = gross_revenue * daily_diminishing_factor
profit = eligible_revenue - ai_operating_cost - overtime_cost - allocated_project_cost
```

当地日前 5 个有效根任务 100%，第 6—10 个 50%，之后 20%；只影响虚拟收入。一个 settlement_unit_key 最多一条成功结算；账本追加、规则版本化、更正使用冲正、每日余额重建对账。经济异常可延迟发虚拟货币，但不得延迟 AI 结果。

## 9. 现实时间、加班和召回

所有时刻以 PostgreSQL `timestamptz` 存储，内部 UTC；公司保存 IANA 时区。设备时间只用于显示。时区变化审计，不重算历史交易。

| 场景 | dispatch_mode | 业务效果 | 2D 表现 |
|---|---|---|---|
| 工作时间 | normal | AI 立即执行 | 员工正常接单 |
| 下班后普通任务 | overtime | AI 立即执行，记录加班成本 | 夜景、亮灯、加班姿态 |
| 下班后紧急任务 | phone_recall | AI 立即执行，记录召回 | 电话姿态、返岗路径 |

`OvertimeSession` 使用 `(employee_id, local_date, rule_version)` 去重；`PhoneRecallEvent` 使用 root_task_id 幂等。Temporal 耐久等待定时器、Signal 和 Activity；PostgreSQL 仍是任务、人事与账本权威。

## 10. 文件、记忆与隐私闭环

首版格式 PDF、DOCX、XLSX、CSV、TXT、PNG、JPEG，普通单文件 25 MB。ZIP、可执行文件、含宏文档和任意嵌套容器不进入直接解析白名单。客户端签名直传隔离对象存储；扩展名、MIME、签名、解压大小、页数和单元格共同校验；解析服务有 CPU、内存、网络与超时限制。

| 数据 | 默认保留 |
|---|---|
| 未保存到项目的原文件 | 任务完成后 30 天 |
| OCR 临时页、缩略图和沙箱中间产物 | 7 天 |
| 保存到项目的原文件 | 直到用户或项目删除 |
| 正式交付物 | 直到用户删除或账号政策到期 |
| 全文 / 向量索引 | 不超过来源生命周期，来源删除后 24 小时内移除在线索引 |
| 原文调试采样 | 默认关闭 |
| 不含原文的业务日志 | 30 天 |
| 匿名聚合指标 | 90 天 |

记忆分任务临时上下文、公司客观知识和员工主观记忆。持久记忆必须有 source_ref、scope、confidence、created_at、expires_at、supersedes 和 deletion_state；无来源摘要不能晋升客观知识；主观记忆默认 90 天衰减。

公司创建时绑定 data_region。文件、结果、记忆和索引默认不跨区；首次向第三方 AI 发送数据前显示接收方、数据类型、用途、区域和保留并取得明确同意。无合规候选时能力不可用，不静默换区。外部文件与网页始终是不可信数据，不能通过自然语言改变工具权限。

## 11. 服务端、HR 与安全结论保持不变

新的客户端决策不删除或弱化以下服务端闭环：

- 模块化单体 + 独立 Temporal、AI、文件、通知和定时工作器；
- PostgreSQL 权威任务、人事、权益、账本、审计与幂等，Redis 只做非权威缓存和限流；
- Outbox 与事件去重，REST 完整快照 + WebSocket 增量 + 推送唤醒；
- 招聘、入职、考勤、请假、加班、绩效、调岗、离职、解雇和任务交接；
- 追加式 FinanceTransaction、TaskSettlement 唯一键、工资应付与付款分离、冲正与每日对账；
- OIDC / OAuth、短期令牌、安全存储、租户校验、RLS、对象签名 URL 和最小权限；
- Prompt Injection、SSRF、文件攻击、工具越权、支付重放、删除不完整和供应链威胁模型；
- 数据导出与耐久删除工作流、备份、时间点恢复和季度恢复演练；
- OpenTelemetry 端到端追踪、成本和供应商账单对账、P0 / P1 告警与排障手册。

客户端场景仍然只是服务端真实事实的投影。场景的姿态结束、用户点击或视觉确认均不能替代任务完成、人事生效、发薪、资产购买或结算事务。

## 12. 实施决策与剩余风险

### 12.1 可直接进入实施

- React Native 0.86.x + TypeScript + New Architecture + Hermes 单客户端；
- 顶级导航：办公室、任务、公司、汇报、我的；办公室默认首屏；
- View / Image / Pressable 分层 2D 场景为默认；
- Reanimated 4 + Gesture Handler 的具体兼容补丁在 PoC 后精确锁定；
- 固定六层、显式 zIndex、OfficeSlot、OfficePath、EmployeeSceneState、SceneSnapshot、visual_intent；
- 全景 4—6 人重点、楼层最多 8 人、同时移动最多 3 人；
- 背景使用双平台验证的 JPEG / PNG / 静态 WebP，透明层优先 PNG；
- 复杂动作局部使用 Rive 或 Sprite Atlas，不依赖动画 WebP 单一路径；
- 普通 RN 压力失败后才条件迁移 actors-and-props 到 Skia Atlas；
- 原生 UI 承担文字、标签、气泡、Sheet、结果、文件、支付和无障碍；
- output.ready → 场景汇报演出 → openResult → 原生结果页 → 快照恢复；
- 黄金视觉签字后才批量生产；
- IAP、多供应商 AI、成本预占、双账本、Temporal、PostgreSQL、文件、HR、经济、安全和隐私方案保持实施。

### 12.2 不能只靠查资料消除的项目

| 剩余项 | 原因 | 决策机制 |
|---|---|---|
| 8 人 / 3 动画真机帧时间 | 依赖真实组件树、图片和设备 | Release 20 分钟压力 Gate |
| 场景增量内存 | 依赖图片解码与平台 | 分平台基线与目标 ≤120 MB |
| 具体依赖补丁 | RN 0.86 组合需实测 | 双平台 PoC 后锁文件冻结 |
| 主观视觉是否成立 | 不能由 API 文档证明 | 黄金视觉双平台书面签字 |
| 资源商业授权 | 取决于采购与合同 | license manifest 与人工复核 |
| 真实模型成本 | 依赖任务混合、重试和缓存 | 评测集、灰度 P50 / P75 / P95 与账单对账 |
| 用户是否理解额度与汇报 | 依赖文案和真实交互 | 购买页 / 首任务 / 结果页用户测试 |
| App Store 最终审核 | 个案性且条款会变化 | IAP 演示、恢复购买、同意与提审前复核 |
| 首发区域 | 市场、法务和采购共同决定 | 公司区域锁定，未批准不接不兼容供应商 |
| 虚拟经济数值 | 依赖用户行为 | 幂等规则固定，数值版本化内测 |

这些项目都已有样本、门槛、负责人和失败动作，不再属于“架构未知”。

## 13. 分阶段 Gate

### Gate -1：发行、安全与成本

冻结首发地区、IAP 商品、第三方 AI 同意、主备供应商、KMS、成本预占、双账本、账单对账、数据区域、团队与预算。失败时不开始批量视觉资源。

### Gate 0A：黄金视觉与核心体验

交付一个竖屏办公室、3—4 房间、4 正式员工、6 姿态、日夜、全景 / 聚焦和一个真实任务。验证用户能理解接单、工作、汇报和原生结果；验证资源授权、构图、遮挡、标签、动态字体和减少动态效果。未签字不量产。

### Gate 0B：普通 RN 压力

在 iOS / Android Release 真机运行固定 20 分钟夹具，达到 P95 UI 帧时间 ≤20 ms、>100 ms 非加载帧 ≤1 / 分钟、场景增量内存接近 ≤120 MB，并完成 50 次前后台无状态错乱。未通过先做两轮优化；确认角色绘制为瓶颈后才迁移 Skia Atlas 层。

### Gate 1：最薄垂直闭环

登录 / 公司初始化 → 真实 AI 任务 → visual_intent → RN 员工接单与工作 → 汇报槽位 → 原生 WorkProduct → 幂等结算 → 资产变化 → 后台恢复。必须在真机反复完成并可追踪，不接受静态演示或手改数据库。

### Gate 2：P0 扩展

扩展三类任务、文件、IAP、权益、成本、3 名核心员工、秘书、HR、工资、资产、项目、知识、导出、删除和无障碍。每个领域先完成状态、失败、权限、幂等、指标和测试，再增加演出。

### Gate 3：发布加固

冻结内容，完成设备矩阵、AI 评测、主备切换、预算降级、数据恢复、删除、账本对账、安全、商店、内测、灰度和服务 / App / 资源回滚演练。

## 15. 最终结论

旧的双运行时客户端结论已由新产品决策正式推翻，不再作为当前方案、低端方案或备用方案。最终路线是 React Native 0.86.x 单客户端：普通 RN 分层 2D 办公室为默认，Reanimated 4 与 Gesture Handler 承担 UI 线程动画和手势，Rive / Sprite Atlas 只用于少量复杂动作，Skia Atlas 只在固定压力 Gate 失败后替换角色与任务道具绘制层。

这一路线的工程信心 **>95%**，因为范围、数据模型、层级、并发、性能门槛、资源校验、恢复和条件式优化均已明确。仍不宣称未经验证的视觉和性能必然成功：主观美术由黄金视觉签字控制，性能由双平台 Release 真机控制。服务端 AI、IAP、成本、Temporal、PostgreSQL、安全、文件、HR、经济、时间、隐私和账本闭环继续保持，不因客户端简化而删减。
