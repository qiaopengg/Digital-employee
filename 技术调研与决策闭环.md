## 1. 调研目标与置信口径

本文针对前期技术设计中的九类不确定性进行闭环：

1. 固定前方俯视角下，是否还需要 Unity；
2. React Native 与 Unity 在手机上的生命周期和内存风险；
3. 办公室扩建后如何保持居中、可读和可点击；
4. 同屏员工、动画、灯光和资源的移动性能；
5. 三类 AI 任务的质量、延迟、成本和供应商可替换性；
6. 员工人格的长期一致性与事实不被改写；
7. 虚拟经济如何奖励真实使用而不鼓励刷消息；
8. 现实时间、加班、打电话、时区和离线恢复；
9. 文件处理、记忆、数据区域和保留删除。

“90% 以上信心”不表示未做实验就声称性能已达标。本文使用以下工程口径：

- 有明确且可实施的首选方案；
- 方案符合官方支持边界，不建立在隐含能力上；
- 有可量化的验收门槛；
- 失败时有不伤害核心 AI 能力的降级方案；
- 决策可逆，资产和业务状态不与单一引擎或供应商锁死；
- 剩余未知被压缩为 PoC、真机或商务合同能回答的问题。

### 1.1 闭环结果

| 事项 | 调研后结论 | 工程信心 | 剩余证据 |
|---|---|---:|---|
| 渲染路线 | React Native 业务壳 + Unity 全屏真 3D | 92% | 双平台真机 PoC |
| 混合生命周期 | 单 Unity Runtime、惰加载、暂停优先、不频繁卸载 | 90% | 连续 50 次切换与内存快照 |
| 固定镜头与扩建 | 正交 40° 候选、单层居中、超限增加楼层 | 95% | 3 种屏幕比例可读性测试 |
| 移动性能 | 30 FPS、8—12 主要员工、三档质量与热降级 | 90% | 20 分钟 soak test |
| AI 执行架构 | IAP 订阅 + 平台托管多供应商 + 原子成本预占 + 双账本 + 版本化评估 | 93% | 真实任务评测、供应商合同、订阅转化和账单对账 |
| 当前 DeepSeek PoC | 官方 V4 文本模型 + 独立 Adapter + 服务端密钥 + 真实错误与流式处理 | 93% | 真实 Key 的低成本契约测试、服务条款与数据区域复核 |
| 员工结果呈现 | Unity 汇报演出 + React Native 员工汇报页 + 暂停 / 快照恢复 | 95% | 交互原型与双平台真机可用性测试 |
| 人格与记忆 | 事实产物与人格汇报两层，记忆带来源与期限 | 94% | 角色回归评测 |
| 虚拟经济 | 按有效根任务结算，修订不重复发放，有软上限 | 93% | 内测数值平衡 |
| 现实时间 | 服务端 UTC + IANA 时区 + Temporal 定时器 | 95% | 夏令时与跨日属性测试 |
| 文件与隐私 | 隔离解析、租户强隔离、来源级删除、区域锁定 | 92% | 首发地区与合同条款 |

上述比例是技术准备度评估，不是统计概率。对尚未真机或内测的项目，“已有高信心解决路径”不等于“已通过发布验收”。

## 2. 渲染引擎决策

### 2.1 硬门槛

本项目的办公室不只是背景图，还必须支持：

- 房间、工位、家具和设备扩建；
- 员工跨房间移动、寻路、坐下和道具交接；
- 共享骨骼、动画分层、IK、LOD 和遮挡；
- 昼夜、灯光、房间状态和资源热更新；
- iOS 与 Android 同一套场景逻辑；
- 场景失败后业务 UI 仍完全可用。

### 2.2 方案比较

| 方案 | 手机嵌入 | 真 3D / 寻路 | 骨骼与资产管线 | 运行成本 | 结论 |
|---|---|---|---|---|---|
| Unity as a Library | iOS / Android 官方支持，但仅全屏 | 完整 | 成熟 | 内存偏高 | 通过功能硬门槛 |
| React Native Skia | 与 RN 紧密集成 | 需自研伪 3D、深度排序和寻路 | 主要面向 2D 图集 | 低 | 适合低档精灵降级，不作主场景 |
| Rive React Native | 官方运行时可用 | 2D 画板和状态机 | 局部角色动效良好 | 低 | 适合空状态和局部动效，不作可扩建办公室 |
| Godot | Android 可作库 | 完整 | 可用 | 中 | 其跨平台 LibGodot 在移动端仍非成熟官方基线 |
| Unreal Engine | 真 3D 能力强 | 完整 | 成熟 | 高 | 对本项目的风格化低多边形场景过重 |

Skia 官方 `Atlas` 能高效渲染大量精灵，Rive 也有状态机和 React Native 运行时，但两者不会自动提供模块化 3D 房间、NavMesh、空间遮挡、骨骼 LOD 和灯光烘焙。选择它们作为主场景，会用更低的引擎内存换取一套自研场景运行时，不符合首版风险最小化。

### 2.3 最终决策

1. 保留 Unity 6.3 LTS + URP 作为真 3D 办公室基线；
2. Unity 只是全屏办公室路由，不承载长文本、设置、文件和支付；
3. 每个 App 进程只有一个 Unity Runtime；
4. 首次进入时惰加载，普通离开时暂停，不每次卸载；
5. 内存压力下先释放房间、高 LOD、Lightmap 和远程包，再考虑 `Application.Unload`；
6. iOS 会话中不调用 `Application.Quit` 后尝试重载；
7. Skia 精灵概览作为 Low 档或 Unity 失败时的可视化降级，React Native 列表作为最终保底。

Unity 官方明确说明：Unity as a Library 在 iOS / Android 仅支持全屏渲染，一个进程只能有一个 Runtime，`Application.Unload` 后仍会保留约 80—180 MB，iOS 完全退出后当前会话不能重新加载。本方案将这些限制当作架构边界，而不是上线后再修补。

### 2.4 退出条件

如果阶段 0 中任一平台在完成两轮定向优化后仍无法满足以下门槛，停止扩大 Unity 投入：

- 4 GB 内存基准机稳态办公室总内存不高于 550 MB；
- 4 房间、12 主要角色在 20 分钟运行后仍稳定 30 FPS；
- 连续进出 50 次无崩溃、黑屏、音频占用或状态错乱；
- 进入后台、锁屏和内存压力后能通过服务端快照恢复；
- Unity 失效不阻断任务、结果、人事和财务。

退出后采用“Skia 分层办公室 + Rive 角色动画 + React Native 业务页”，并同时取消自由家具摆放、真实 NavMesh、多人精确座位和复杂灯光，避免在 2.5D 中重写 Unity。

## 3. 固定镜头与办公室扩建

### 3.1 镜头冻结值

- 默认使用正交摄像机，不使用透视相机作为首版基线；
- 摄像机位于办公区前上方，无横向偏航，无滚转；
- 初始俯视角为 40°，PoC 只比较 35° / 40° / 45° 后冻结；
- 默认不允许自由旋转，聚焦也不改变朝向；
- 办公区默认容纳于可用安全区约 72% 宽、62% 高的中心框；
- 角色名称、状态和热区按屏幕空间布局，不直接依赖模型碰撞体大小；
- 前墙、屋顶、高柜使用镜头截面或透明淡出，不依赖旋转解遮挡。

40° 是内容生产的候选默认值，不是跳过原型的永久美术结论。原型仅允许在三个离散俯角中选择，避免建模期持续漂移。

### 3.2 扩建不采用无限画布

办公室采用“单层主场景 + 楼层切换”：

```text
公司大楼
├─ 当前活动楼层：屏幕中心真 3D，最多 6 个功能房间
├─ 其他已解锁楼层：楼层选择器 + 摘要状态
├─ 老板、会议、档案等特殊房间：使用稳定功能槽位
└─ 超额员工：分配到其他楼层或以团队摘要表现
```

规则：

- 每层最多 6 个主房间、8—12 名完整动画员工；
- 扩建优先解锁新楼层或新功能槽位，不持续缩小现有房间；
- 每层切换时重新用 `office_content_bounds` 适配中心框；
- 非当前楼层不运行 Animator、NavMeshAgent、IK 和灯光，只保留服务端状态；
- 跨层移动表现为离开当前层和进入目标层的两个可中断意图，不实时模拟电梯；
- 用户点击秘书摘要可查看所有楼层任务，不必逐层寻找。

这一设计同时解决了固定视角、场景居中、员工增长、小屏可读性和性能上限。

## 4. 移动性能和生命周期闭环

### 4.1 支持与体验分层

技术支持与 3D 高质量体验不使用同一最低线：

| 层级 | 技术建议 | 办公室体验 |
|---|---|---|
| 业务最低支持 | React Native 当前最低目标为 iOS 15.1+ / Android API 24+ | 任务、结果、人事、财务和列表办公室必须可用 |
| 3D Low | 64 位、有效内存低或热状态较差 | 30 FPS、最多 6—8 主要员工、低分辨率、无附加阴影 |
| 3D Medium | 4 GB 及以上物理内存的基准档 | 30 FPS、8—12 主要员工、标准细节 |
| 3D High | 高性能且热状态充足 | 可选 60 FPS 和更高画质，不增加业务能力 |

首次进入 3D 前使用设备信息设定保守初值，再根据真实帧时间、内存压力和热状态只向下调整。一次会话内不频繁上下震荡。

### 4.2 已选渲染预设

- URP Forward；
- 单主摄像机；
- 静态房间使用 Baked Lit，员工使用 Simple Lit；
- 关闭 HDR、Depth Texture、Opaque Texture、SSAO、运动模糊和景深；
- 一盏主方向光，Low 档关闭实时阴影；
- 静态 Lightmap，房间灯光使用分组烘焙变体；
- SRP Batcher、Shader Variant 剥离、少透明材质；
- ASTC 作为现代设备主贴图压缩，平台导入设置固化；
- 30 FPS 档位帧预算 33.3 ms，内部目标 P95 不高于 30 ms；
- 无交互且无关键动画时，NPC 逻辑降频，不变更真实任务状态。

Unity 官方 URP 建议关闭不必要的 Depth / Opaque Texture、HDR、附加灯阴影和后处理，减少摄像机，启用 SRP Batcher，并在低端移动设备上使用 Baked Lit / Simple Lit。这些直接成为本项目预设，不留给开发者任意选择。

### 4.3 量化验收

| 项目 | 阶段 0 门槛 |
|---|---|
| Unity 首次进入 | 基准设备 P95 ≤ 6 秒，期间 RN 摘要可用 |
| Unity 暖进入 | P95 ≤ 2 秒 |
| 稳态帧率 | 20 分钟后 P95 帧时间 ≤ 33.3 ms |
| 严重卡顿 | 超过 100 ms 的非加载帧 ≤ 1 / 分钟 |
| 稳态内存 | 4 GB 基准机 ≤ 550 MB，以平台工具为准 |
| 切换泄漏 | 50 次进出后稳态内存较第 5 次增长 < 5% |
| 热状态 | 20 分钟内不持续处于平台严重 / 危急档；一旦进入立即降档 |
| 生命周期 | 50 次前后台 / 锁屏 / RN 切换无崩溃与状态错乱 |
| 快照恢复 | 丢失任意本地动画队列后，5 秒内回到服务端当前状态 |

编辑器、模拟器和 Development Build 不用于通过性能门槛。必须使用 Release / Shipping 真机包，同时保存 Unity Profiler、Memory Profiler、Xcode 或 Android 平台记录。

## 5. 美术与动画产能方案

### 5.1 黄金资产优先

批量生产前只制作一组完整“黄金资产”：

- 1 套标准 Humanoid 骨骼；
- 2 个体型变体，但骨骼结构不变；
- 2 个核心员工外观；
- 1 套通用办公服、发型、鞋和表情色板；
- 1 个办公区、1 个会议室、1 个老板室、1 个档案 / 设备房；
- 24 个核心 Clip：Idle 变体、Walk、坐下、起身、打字、阅读、书写、思考、接听、记录、挂断、交接、汇报、会议、检查、喝水、下班、加班疲惫等；
- 每个角色 LOD0 / LOD1 / 远距离简化表示；
- 一套 Blender 导出预设、Unity Import Preset 和自动校验。

只有黄金资产在固定镜头、动画重定向、道具挂点、遮挡、LOD、Addressables 和真机性能上全部通过，才允许复制生产。

### 5.2 产能控制

- 角色差异优先来自头部、发型、服装组合、材质色板和动作选择，不复制骨骼；
- 人格差异由同语义 Clip 变体、播放速度、停顿和小动作组合，不为每位员工重做全套动画；
- 房间使用模块化地板、墙、门、工位和家具，每个新房间只允许少量特有资产；
- 所有资产使用脚本化 FBX 导出，禁止手工调整轴向、缩放和 Clip 命名；
- 发布前自动检查多边形、材质数、骨骼、贴图、挂点、LOD、动画事件和 Addressable 依赖；
- 内容扩展速度以“通过自动校验的有效资产 / 人周”衡量，不以 Blender 源文件数量衡量。

Blender 的 FBX 导出器可烘焙修改器和动画，但骨骼轴向、Action 关联和材质转换存在已知注意事项，因此本项目使用确定性导出脚本和 Unity Import Preset，而不依赖美术人员每次手动记住参数。

## 6. AI 质量、成本与人格闭环

### 6.1 不在架构阶段猜一个永久最佳模型

模型质量、价格、限额、数据政策和区域都会变化。产品现已确认采用“App Store IAP 自动续期订阅 + 平台托管多供应商 AI”：用户不提供 API Key，也不直接购买某家模型的 Token；平台采购供应商能力，以订阅净收入承担模型成本，并通过订阅周期算力额度控制单用户支出。首版冻结一个主供应商和一个经过生产验证的备用供应商，不在客户端永久写死具体型号，也不允许普通用户添加自定义端点。

AI Gateway 使用以下中立能力协议：

```text
capability_profile
input_references[]
required_output_schema
tool_policy
safety_context
data_region
retention_class
latency_budget
usage_budget
entitlement_cycle_id
reservation_id
model_price_version_id
```

供应商适配层负责把这些要求映射为具体 API，业务代码不直接依赖某个供应商的会话 ID、文件 URI 或工具命名。官方协议调研确认 OpenAI Responses、Anthropic Messages 与 Google Gemini `generateContent` 在端点、流式事件、工具、结构化输出、用量和错误语义上存在实质差异；首版使用三种原生协议或首发地区的等价协议证明统一能力层，另保留严格定义、经过实测且能力受限的 OpenAI-Compatible 最小协议，不能把基础聊天兼容等同于完整任务兼容。

平台后台保存按环境和数据区域隔离的 `AIProviderConfiguration`，为任务类型配置轻量、均衡和高阶模型路线。自动故障切换只允许进入已经通过能力评测、价格有效、健康正常、地区兼容且用户明确同意的备用路线；没有合法候选时进入可恢复阻塞，不能切换到未验证模型或未披露供应商。

### 6.2 模型分层路由

| 层级 | 任务 | 约束 |
|---|---|---|
| 快速结构层 | 意图、缺失信息、任务分类、秘书摘要 | 必须输出 Schema，不产生最终事实结论 |
| 核心交付层 | 写作、调研、文件 / 数据分析 | 使用平台已验证路线；按任务难度、质量、预算和地区选择模型并强制来源与中间结构 |
| 验证层 | 要求覆盖、引用、计算、安全和输出结构 | 优先确定性代码，模型复核不代替规则 |
| 人格表达层 | 员工短汇报、进度描述、引导语 | 只读 `canonical_output` 的事实摘要，不可新增数字、来源和结论 |

核心交付层输出版本化 `WorkProduct`：

```text
answer_blocks[]
claims[] { claim_id, text, source_ids[], confidence }
numbers[] { value, unit, source_or_formula }
citations[]
limitations[]
unresolved_questions[]
artifact_refs[]
```

人格层只生成 `PersonaReport`，服务端校验它引用的 `claim_id` 必须存在，数字与来源不得出现新集合。失败时直接回退到标准汇报模板，不阻塞正式结果。

### 6.3 评测与发布门槛

首版建立不少于 450 条版本化样本，三类任务各不少于 150 条；其中至少 20% 为对抗 / 边界样本，20% 含文件、图片或表格，20% 为多轮修订。

| 指标 | 发布门槛 |
|---|---|
| 用户要求覆盖 | 人工双评平均 ≥ 4.2 / 5 |
| 引用支持率 | 可核查事实声明有支持来源 ≥ 95% |
| 引用错配 | 严重错配 < 1% |
| 文件抽取 | 关键字段准确率 ≥ 98%，表格数值必须程序复核 |
| 人格事实漂移 | 数字、来源、限制和结论的严重改写 = 0 |
| 工具越权 | 自动与对抗集中成功越权 = 0 |
| 可恢复性 | 限流、超时和部分工具失败可恢复或清晰降级 ≥ 99% |

自动指标只用于预筛，发布样本必须人工盲评。模型、Prompt、检索、工具或角色模板任一变更都重跑对应切片。NIST 生成式 AI 风险管理资料建议在发布前建立可审查的评估阈值，并在上线后持续评估；本门槛将其转化为项目级的 go / no-go 规则。

### 6.4 成本控制

- 每个订阅产品定义一个订阅周期 AI 算力额度；这是平台内部成本预算，不是可提现余额、供应商 Token 或虚拟公司资金；
- 成本基数采用扣除 Apple 佣金和适用税费后的订阅净收入。模型成本目标为 P75 用户不超过净收入的 25%，设计硬上限为 30%；最终价格和额度必须以真实任务混合与 App Store 实际结算验证；
- 每次调用先按 `ModelPriceVersion`、预计输入 / 输出、缓存和工具费用计算 `estimated_cost_microunits`，在同一数据库事务内原子创建 `UsageReservation`；余额不足时先换用满足质量门槛的低成本路线，否则在调用前阻止；
- 调用结束按供应商实际用量结算，写入 `ProviderUsageLedgerEntry`，再以同一幂等键写入 `UserAllowanceLedgerEntry`；实际成本低于预占时释放差额，高于预占时只在已批准容差内补扣；失败、超时和重试不能重复计费；
- 同时设置用户订阅周期、单任务、供应商日 / 月和平台全局硬上限。达到 80% 提醒，95% 进入受控降级，100% 停止产生新成本；
- 文件先解析与定位，只向模型提供任务所需片段；
- 重复使用的公司规则、文档和长上下文可使用经评估的供应商缓存，但必须与保留政策兼容；
- 超过任务预算前先缩小无关上下文或切换已验证的低成本路线；仍无法完成时说明额度原因，不静默截断结果；
- 同一文件只解析一次，派生任务引用版本化解析产物；
- 后台按用户、订阅产品、任务类型、供应商、模型、价格版本和时间窗口统计请求、Token、缓存、工具、重试、预占与实际成本；每日对供应商账单和内部成本账本进行对账，差异超过 3% 告警并暂停扩大流量。

### 6.5 IAP、平台密钥与 App Store 边界

- App 内持续数字服务只通过 App Store IAP 自动续期订阅销售；首版不提供 BYOK、外部付款入口、网页购买引导或外部购买价差提示；
- 客户端使用 StoreKit 2 发起购买和恢复购买，并携带稳定、不可逆推出账号身份的 `appAccountToken`；服务端通过 App Store Server Notifications V2 和 Server API 维护权威订阅状态、退款、撤销、宽限期和过期；
- 平台供应商 Key 写入 KMS / Secret Manager，按 Local / Development / Staging / Production、供应商和数据区域隔离；普通数据库只保存凭据引用；
- 只有 AI Worker 的最小权限服务身份可在执行时取得平台 Key；Key 不进入客户端缓存、Unity、Temporal History、日志、分析、客服后台和崩溃报告；
- 首次向每个第三方 AI 发送问题、文件片段或公司知识前，必须展示接收方、数据范围和用途并取得明确同意；
- 平台运营后台可以测试、轮换、停用供应商配置和启动成本熔断，但不能显示完整密钥；所有操作写审计日志；
- App Store 审核使用沙盒 IAP、专用审核账号和可重复的演示任务，必须展示订阅条款、额度规则、恢复购买、订阅管理、隐私政策及第三方 AI 数据同意。

### 6.6 当前 APP 优先 PoC 的 DeepSeek 接入结论

当前验证阶段先接 DeepSeek，不改变生产多供应商架构。适配器使用官方 `https://api.deepseek.com`，以 `/models` 结果和后台允许清单为准；文档更新时间的候选文本模型为 `deepseek-v4-flash` 与 `deepseek-v4-pro`。官方已为旧别名 `deepseek-chat`、`deepseek-reasoner` 给出下线窗口，因此代码、配置和测试不得依赖旧别名。

实现闭环如下：

- Key 只存在于本地 / 测试服务端环境或 Secret Manager；客户端只访问项目服务端，不能直连供应商；
- Adapter 解析普通响应和 SSE，忽略空事件与 keep-alive，支持取消，并把认证失败、供应商余额不足、限流、内部错误和过载映射为稳定平台错误；
- JSON Output、Tool Calls 和用量字段分别做契约测试，不因基础 Chat Completions 可用就默认所有能力可用；
- DeepSeek 接口按无状态多轮调用处理，平台持久化、裁剪和重建上下文；`user_id` 使用不可逆假名；
- 当前模型基线按文本输入处理；文件、图片与表格先由受控 OCR / 解析器转成带页码、工作表或单元格来源的文本；
- PoC 使用合成或非敏感数据。官方条款和隐私政策说明的下游告知、同意、数据权利与处理区域责任进入生产发布门禁。

结论：对“完成一条真实 DeepSeek 文本任务并可安全恢复”的工程方案信心为 93%。剩余不确定性是实际账号限额、实时价格 / 模型变化以及目标地区的法律与合同适用性；这些必须以配置查询、契约测试和发布审查回答，不能在客户端代码中写死。

### 6.7 AI 结果如何在办公室场景中交付

Unity as a Library 在 iOS / Android 只支持全屏并且同一进程只有一个 Runtime；卸载后仍可能保留约 80—180 MB，iOS 会话中完全退出后不能再次加载。因此把完整长答案、表格、代码和文件预览放进 Unity，会同时放大排版、复制、无障碍、内存和生命周期风险。

最终方案是“空间汇报 + 正式成果”双层体验：

```text
output.ready
→ 员工整理文件并走到老板汇报点
→ 用户点击查看汇报
→ Unity 发出 openResult(task_id)
→ React Native 暂停 Unity 并打开员工汇报页
→ 展示 PersonaReport + 权威 WorkProduct + 来源 / 限制 / 操作
→ 返回时按服务端快照恢复办公室
```

这不是把沉浸感移除，而是把人格化的“谁来交付”留在 3D，把生产力的“结果能否真正使用”交给原生业务 UI。多个完成任务由秘书汇报收件箱聚合；Unity 失效时同一收件箱和办公室摘要仍可打开结果。未知持续时间的模型任务只显示持续活动和已发生的真实阶段，不使用虚构百分比；这与 Apple 对不可量化任务使用不确定进度、允许用户继续做其他事的 HIG 建议一致。

结论：该导航与生命周期方案的工程信心为 95%。仍需通过原型验证用户是否理解“员工已完成—点击看正式报告”，并在真机确认结果页停留期间 Unity 的渲染、音频和 NPC 更新确实停止。

## 7. 虚拟经济与防刷闭环

### 7.1 从“每条消息奖励”改为“有效根任务结算”

经济系统使用 `root_task_id` 作为主结算单元：

- 首次成功交付产生一次任务收入；
- 该任务内的追问、补充资料、返回修改和重试不重复发放基础收入；
- 只有用户明确新建、有独立目标和新 `root_task_id` 的任务才是新结算单元；
- 系统拒绝、空结果、用户在执行前取消、解析失败且无交付物时不产生收入；
- 系统故障恢复后仍然对同一 `settlement_unit_key` 结算，不因重试多发收入；
- 虚拟收入永远不影响用户已购订阅权益或订阅周期算力额度，也不可兑换成真实货币、App Store 权益或供应商模型额度。

### 7.2 首版算法

```text
gross_revenue = base_value(task_value_class)
eligible_revenue = gross_revenue * daily_diminishing_factor
profit = eligible_revenue - ai_operating_cost - overtime_cost - allocated_project_cost
```

`task_value_class` 在执行前由确定性规则根据任务类型、文件量、要求产物和工具计划分类，不由用户自报，也不直接按 token 越多收入越高。

首版软上限建议：

- 当地日前 5 个有效根任务：100% 收入；
- 第 6—10 个：50% 收入；
- 第 11 个及以后：20% 收入；
- 软上限只影响游戏收入，不限制 AI 回答和用户已购权益；
- 数值通过版本化 EconomyRule 配置，必须经过内测模拟后才能调整。

明确的完全重复提交可以不产生新收入；语义相似检测只用于风险标记和限速，不单独作为扣除收入依据，避免误伤真实修订。

### 7.3 完整性不变量

- 一个 `settlement_unit_key` 最多一条成功结算；
- 账本中的每笔借贷或收支必须平衡；
- 所有规则记录 `economy_rule_version`；
- 已结算数额不随新规则回溯改写；
- 更正使用冲正与新交易，不修改旧交易；
- 每日运行账本重建与余额快照对账；
- 经济反异常失败时，延迟发放虚拟货币，但不延迟 AI 结果。
- 虚拟公司资金、订阅权益和 AI 算力额度为三个独立账户，彼此不能兑换、抵扣或混入同一余额。

## 8. 现实时间、加班和电话

### 8.1 时间权威

- 所有时刻以服务端 `timestamptz` 存储，内部统一 UTC；
- 公司单独存储 IANA 时区名称，不只存 UTC 偏移；
- 排班、工资日和当地日由服务端日历库计算；
- 设备时间只用于 UI 显示输入，不作为发薪、考勤和收入依据；
- 时区变更创建审计记录，不重算已发生的交易。

PostgreSQL `timestamp with time zone` 在内部按 UTC 存储，IANA 完整时区名称包含夏令时规则；只保存数字偏移无法处理季节性变化。

### 8.2 非工作时间派工

员工不拒绝用户任务。提交时根据服务端公司时间生成：

| 场景 | `dispatch_mode` | 业务效果 | 演出 |
|---|---|---|---|
| 工作时间 | `normal` | AI 立即执行 | 员工正常接单 |
| 下班后普通任务 | `overtime` | AI 立即执行，记录加班成本 | 房间亮灯、员工返岗 / 留岗 |
| 下班后紧急任务 | `phone_recall` | AI 立即执行，记录召回或紧急费 | 打电话、接听、返岗 |

非工作时间默认预选“加班处理”，用户可一键切换“打电话紧急召回”。这一选择只决定演出、虚拟成本和人事记录，不人为推迟底层 AI。

### 8.3 工作流

```text
submit_task
→ 服务端确定 company_local_time 与 dispatch_mode
→ 写入 TaskDispatch、幂等键与 Outbox
→ 立即启动 AI Activity
→ 并行写入 OvertimeSession 或 PhoneRecallEvent
→ 场景投影生成打电话 / 亮灯 / 返岗意图
→ 结果完成后独立结算任务收入与加班成本
```

`OvertimeSession` 使用 `(employee_id, company_local_date, overtime_rule_version)` 去重，避免同一晚每个追问都重复计一次到岗费；`PhoneRecallEvent` 使用 `root_task_id` 幂等，修订和重试不重复召回。

Temporal Workflow 可以耐久等待定时器、Signal 与 Activity，在工作器故障后通过历史重放恢复。但 PostgreSQL 仍是任务、人事和账本权威，Temporal 历史不代替业务账本。

## 9. 文件、记忆与隐私闭环

### 9.1 首版文件基线

- 格式：PDF、DOCX、XLSX、CSV、TXT、PNG、JPEG；
- 普通单文件上限：25 MB；
- 超过上限的文件不直接失败，进入明确的大文件评估或引导拆分；
- ZIP、可执行文件、含宏 Office 文件和任意嵌套容器不进入首版直接解析白名单；
- 客户端直传隔离对象存储，解析服务不与 API 主进程共享网络权限；
- 扩展名、MIME、文件签名、解压后大小和页 / 单元格数共同校验；
- 原文件不作为公开 CDN 对象，下载必须重新检查成员权限。

OWASP 文件上传指南明确建议使用扩展名白名单、不信任 `Content-Type`、重命名、限制大小、隔离存储、杀毒 / 沙箱和纵深防御。本基线将这些控制转为默认流水线。

### 9.2 保留默认值

| 数据 | 默认保留 | 备注 |
|---|---|---|
| 未保存到项目的原文件 | 任务完成后 30 天 | 用户可提前删除 |
| 沙箱中间件、OCR 临时页和缩略图 | 7 天 | 可根据任务状态延长，有上限 |
| 明确保存到项目的原文件 | 直到用户删除、项目删除或账号政策到期 | 在 UI 明确标识 |
| 正式交付物 | 直到用户删除或账号政策到期 | 可导出 |
| 全文 / 向量索引 | 不超过来源生命周期 | 来源失效后 24 小时内移除在线索引 |
| 原文调试采样 | 默认关闭 | 开启需受控、脱敏与自动过期 |
| 业务日志 | 30 天 | 不含用户原文 |
| 匿名聚合指标 | 90 天 | 不可还原任务内容 |
| 在线系统删除任务 | 24 小时内完成 | 备份按不可变保留窗口自然过期，不用于普通恢复 |

当地法律、企业合同或供应商能力要求更短时，只能缩短上述默认期限，不能静默延长。

### 9.3 记忆不是无条件保存对话

| 记忆层 | 内容 | 权威 | 默认期限 |
|---|---|---|---|
| 任务临时上下文 | 当前任务必需的对话、文件片段 | 仅当前任务 | 跟随任务保留 |
| 公司客观知识 | 用户明确保存的规则、品牌、项目事实 | 可进入正式交付 | 直到删除或设定到期 |
| 员工主观记忆 | 偏好、交往和可以变化的印象 | 只影响口吻与演出 | 默认 90 天衰减，可显式保留 |

每条持久记忆必须包含 `source_ref`、`scope`、`confidence`、`created_at`、`expires_at`、`supersedes`、`deletion_state`。无来源摘要不能自动晋升为公司客观知识。

### 9.4 区域与供应商

- 公司创建时绑定 `data_region`；
- 原文件、解析产物、正式结果、记忆和索引默认不跨区；
- 首次使用第三方 AI 前，用户必须确认供应商、数据类型、用途、数据区域和保留说明；
- 模型路由必须同时满足平台启用状态、同意版本、任务能力、价格有效性和数据区域 / 保留等级；
- 平台已验证路线中没有合规候选时，该能力明确不可用，不静默把数据路由到其他地区或未披露供应商；
- 区域迁移是显式、可审计的导出—导入流程，不直接修改一个字段；
- 已验证供应商清单上线前需要确认 API 接入条款、价格与限流、不训练选项、保留策略、子处理者列表和删除能力；普通用户不能启用未验证连接或自定义端点。

OWASP 明确指出，文件和网页中可以存在间接 Prompt Injection，RAG 与微调不能完全消除该风险。因此本项目始终将外部内容标记为不可信数据，工具权限在确定性代码中强制，不让模型凭自然语言改变权限。

## 10. 实施决策与剩余风险

### 10.1 可直接进入实施的决策

- React Native + Unity 全屏混合客户端；
- 前方高位 40° 候选正交俯视、办公区居中、禁止自由旋转；
- 每层最多 6 个主房间，扩建使用新楼层，不无限缩放；
- 一个 Unity Runtime，惰加载，普通离开只暂停；
- 30 FPS 基线和 8—12 主要角色上限；
- 当前先用一名核心员工、一名秘书和一个 DeepSeek V4 文本任务验证体验，再用 4 个房间、8—12 名占位员工验证压力；
- App Store IAP 自动续期订阅、平台托管多供应商、订阅周期算力额度、原子成本预占、双账本和授权范围内故障切换；
- WorkProduct 与 PersonaReport 分层；
- Unity 负责汇报演出，React Native 员工汇报页负责正式结果；进入结果页暂停 Unity，返回时按服务端快照恢复；
- 按有效 `root_task_id` 结算收入，修订不重复奖励；
- 非工作时间默认加班，可切换电话召回，AI 都立即执行；
- UTC + IANA 时区 + Temporal 定时器 + PostgreSQL 权威事实；
- 文件白名单、25 MB 普通上限、隔离解析和来源级删除。

### 10.2 不能通过继续查资料消除的剩余项

| 剩余项 | 为什么不能纸上消除 | 已定处理 |
|---|---|---|
| 具体手机的峰值内存与热降频 | 依赖真实资产、驱动和机型 | 阶段 0 发布包真机门槛，失败则按退出条件降级 |
| 真实任务的模型成本分布 | 依赖用户任务长度、文件比例、重试率、模型路线和缓存命中 | 用版本化评测集与灰度数据测 P50 / P75 / P95；P75 目标不超过订阅净收入 25%，设计上限 30% |
| 用户是否理解订阅算力额度 | 依赖定价、文案、任务差异和真实使用习惯 | 在购买页、任务前和用量页展示可理解的剩余量与高成本提示，用户测试验证理解率与首任务完成率 |
| P0 主备供应商和模型档位 | 取决于首发地区、价格、质量、数据合同和生产限流 | 冻结一主一备与轻量 / 均衡 / 高阶路线，完成契约、评测、故障和价格版本测试后才启用 |
| App Store 对订阅与 AI 核心流程的最终审核结果 | 审核具有个案性，条款和商品配置可能变化 | 只使用 IAP，提供恢复购买、订阅管理、额度说明、第三方 AI 同意和专用审核演示；提审前复核最新指南 |
| 订阅能否覆盖模型与基础设施总成本 | 取决于 Apple 实际结算、留存、任务混合、KMS、工作流、存储、推送和流量 | 用净收入而非标价建模，分别统计模型成本和平台成本，设置用户、供应商与全局硬上限并按月复盘单位经济 |
| 供应商账单与内部成本是否一致 | 供应商用量字段、缓存计费、批处理折扣、税费和价格调整可能变化 | 价格版本化、原始用量留存、每日账单对账；差异超过 3% 告警并暂停扩大流量 |
| 虚拟经济的最佳数值 | 依赖真实用户行为 | 幂等和防刷规则已定，只在版本化数值上内测调优 |
| 首发数据区域 | 是市场、法务和采购决策 | 架构已支持单公司区域锁定，未批准前不接入不兼容供应商 |

这些项目不再是“不知道怎么解决”，而是已经有明确决策机制、样本、门槛和退出条件的验证项。项目范围、订阅转化、平台密钥、App Store、首发地区、供应商合同、资源和单位经济等非单项技术风险，统一以《项目落地风险与验证计划.md》为开工和发布门禁。

## 12. 结论

调研后，不再建议为“固定视角”盲目改成 2.5D，也不建议忽略 Unity 的内存与嵌入限制。当前最稳妥的路线是：用 Unity 实现空间、角色与扩建，用 React Native 保证生产力、文件、订阅和无障碍，用平台托管的多供应商 AI 保证质量与可替换性，并由服务端统一维护订阅权益、成本预占、双账本、现实时间、虚拟经济与记忆恢复。

这条技术路线本身已经具备超过 90% 的工程准备度，但不能据此推导整个项目已有 90% 的发布成功率。项目仍必须通过《项目落地风险与验证计划.md》的发行、IAP、平台密钥、供应商账单、成本上限、资源、单位经济和内测门禁，以及本文要求的真机、真实任务和真实用户验收。
