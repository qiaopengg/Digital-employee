> 文档状态：落地风险基线 / 开工与发布门禁

> 更新日期：2026-07-21
> 需求基线：《需求设计.md》v0.9
> 技术基线：《技术设计.md》v0.8
> 适用范围：从立项、PoC、内测到 App Store 首次公开发布

## 1. 审查结论

本项目可以落地，但当前不能把“技术方案已经比较完整”误写成“项目已经有 90% 的发布成功概率”。经全量审查需求、技术设计和既有技术调研后，最大的落地风险依次是：

1. **P0 范围与团队、预算、时间尚未形成可执行闭环**；
2. **订阅转化、算力额度理解和模型成本之间尚未被真实用户与真实任务共同证明**；
3. **平台承担 AI 供应商采购、生产密钥、真实账单和持续运营责任**；
4. **首发地区未冻结，App Store 支付规则与中国大陆监管路径存在实质差异**；
5. **“多 AI 供应商”不是换一个 Base URL，多种官方协议、价格、能力和错误语义需要分别验证**；
6. **平台模型路由必须同时平衡答案质量、成本、延迟、地区和故障切换**；
7. **完整 HR、经济、现实时间、任务和轻量场景表现共同形成状态组合爆炸**；
8. **React Native 轻量互动办公室的真机性能、视觉一致性与内容产能仍需实测**；
9. **订阅必须同时覆盖模型 Token、KMS、工作流、文件、推送、存储、带宽和运维成本**；
10. **“公司归属感 + 任务收益”能否显著提高持续使用，仍是需要数据证明的产品假设**。

其中，第 1—4 项在没有通过门禁前，不应进入批量美术资产或全量 P0 开发。R8 必须由“技术性能 Gate”和“黄金视觉及内容产能 Gate”分别给出证据；两者都通过后，才能冻结轻量互动办公室方案并扩大资产生产。

### 1.1 对“90% 以上把握”的准确承诺

本文承诺的是：**把每个重大风险压缩为有负责人、有证据、有量化门槛、有失败降级和有停止条件的工程问题，使通过全部发布门禁后的工程发布准备度达到 90% 以上。**

以下事项不能被诚实地保证为 90% 必然成功：

- 用户一定长期留存或付费；
- Apple 一定一次审核通过；
- 监管部门或第三方供应商未来规则不变化；
- 任意价格或任意模型路线都能同时达到相同质量和成本；
- 在团队、预算、目标日期都未知时仍保证按期完成。

因此采用“关键风险逐项通过、不得平均”的口径：任何一项 P0 门禁未通过，整体准备度就不能宣称达到 90%。

## 3. R1：P0 范围与资源不闭环

### 3.1 为什么它是第一风险

当前 P0 同时包含：三类 AI 任务、文件/OCR/表格、React Native 轻量互动办公室、三名核心员工与秘书、少量普通员工、完整 HR 生命周期、现实时间与加班、经济账本、招聘和扩建、IAP 订阅、平台托管多供应商 AI、成本控制、记忆、后台恢复、账号与删除、无障碍、分析和 App Store 发布。

这些能力单独都可实现，但一起进入首次公开版会形成跨移动端、视觉与动效资产、服务端、AI、安全、支付和运维的长依赖链。当前文档没有冻结实际团队、可用预算和目标日期，任何排期都是假精确。

### 3.2 解决方案：完整闭环不删除，但采用“浅闭环 + 深切片”

用户已确认完整人力资源模拟器有必要，因此不删除招聘到离职的生命周期。首版改用以下实施定义：

- **HR 完整但浅**：招聘、入职、考勤、工资、绩效、晋升、调岗、离职和解雇每个状态都可真实走通，但首版每个环节只保留一套确定性规则和少量事件；
- **核心任务深**：三类任务必须提供真实可用的输入、执行、引用、结果、修订和恢复，不做只有演示效果的空壳；
- **经营完整但少内容**：收入、工资、资产购买，以及临时工作点、小型独立办公室、标准公司楼层三套模板之间的扩建、降级和搬迁走通；首版不再增加第四套场景，也不追求大量商品、岗位、房间和随机事件；
- **先纵向后横向**：先用一名员工、一个任务、一次结算和一项资产完成真实端到端，再扩到全部 P0；
- **轻量场景优先**：先冻结场景分层、语义槽位、预设路径和黄金视觉，不把内容规模当作首版完整度；
- **模块化单体**：不因为领域多就提前拆微服务；PostgreSQL 是业务事实，Temporal 只负责编排。

### 3.3 开工门禁

大规模开发前必须形成一张资源能力表，每项都有姓名、可投入比例和替补方案：

- React Native、Reanimated 与 Gesture Handler；
- iOS / Android Release 构建、真机性能与生命周期；
- 条件式 Skia 角色层、Rive 局部动效与性能诊断；
- 视觉设计、角色/场景插画、动效、资产规范与内容制作；
- 服务端、PostgreSQL 与可靠性；
- AI Adapter、评测和安全；
- QA、真机、隐私和发布；
- 云平台、值班和事故响应；
- 产品、内容、法务与 App Store。

通过条件：

1. 每个 P0 能力有唯一负责人；
2. 用最薄垂直切片实测吞吐后，剩余工作量不超过可用产能的 70%，保留至少 30% 集成、审核和返工余量；
3. 视觉资产必须以“通过规范、真机验收并可直接入库的有效资产/人周”计量，同时预留 30% 返工，不以原稿数量计量；
4. 没有任何关键能力只依赖一个无法替代且没有预留时间的人；
5. 目标日期、团队或预算任一变化时，重新计算，不通过“全员加班”补齐结构性缺口。

### 3.4 停止与收缩条件

- 资源表未通过：只做 Gate 0/1 原型，不启动全量内容生产；
- 排期超载：先减少事件、装饰、商品、复杂 HR 规则和 Android 公开发布范围，不删除三类任务和 HR 生命周期；
- 纵向切片无法稳定完成：停止增加新系统，先修通同一条链路；
- 黄金视觉未由产品负责人明确签字：禁止批量生产角色、房间、姿态和昼夜变体。

## 4. R2：订阅转化、额度理解与模型成本

### 4.1 风险本质

平台托管 AI 消除了普通手机用户申请供应商账号、充值和配置 API Key 的门槛，但平台因此直接承担每一次模型调用的真实费用。风险不再是“用户能否连接模型”，而是三件事能否同时成立：用户愿意通过 IAP 订阅、用户能理解不同任务会消耗不同算力、平台能在不明显伤害答案质量的前提下把成本控制在订阅净收入以内。

只看 App Store 标价、平均 Token 或理论请求数都会低估退款、Apple 佣金、税费、长文件、重试、工具调用和高阶模型带来的尾部成本。

### 4.2 解决方案

- App 内持续数字服务只使用 App Store IAP 自动续期订阅；首版不提供 BYOK、外部付款、网页购买引导或“无限 AI”承诺；
- 每个订阅产品定义一个订阅周期 AI 算力额度，购买页和用量页说明额度是平台服务预算，不是供应商 Token、现金或虚拟公司资金；
- 以 Apple 实际结算净收入为成本基数，不以 App Store 标价计算。模型成本目标为 P75 用户不超过净收入的 25%，设计硬上限为 30%；
- 每次调用执行“估算 → 原子预占 → 调用 → 实际结算 → 释放差额”，余额不足时先缩短无关上下文或使用满足质量门槛的低成本路线，否则在产生费用前阻止；
- 用户额度账本只回答“本订阅周期还能用多少”，供应商成本账本只回答“平台实际欠供应商多少”，两者通过同一 `execution_attempt_id` 对账但不能混成一个余额；
- 同时设置单任务、用户订阅周期、供应商日 / 月和平台全局硬上限；80% 提醒，95% 受控降级，100% 停止产生新成本；
- 免费 / 试用只能使用独立小额度与低成本路线；首版不出售额外算力包，也不出售虚拟货币、办公场景、家具或设备，避免增加 IAP 商品、退款和用户理解复杂度；
- 后台按订阅产品、用户、任务类型、供应商、模型、价格版本和时间窗口展示 P50 / P75 / P95 成本、毛利、预占差异、失败成本与重试成本。

### 4.3 真实用户与成本验证门禁

第一轮使用不少于 24 名目标用户做购买页和额度理解测试；第二轮进入不少于 100 名真实内测用户的漏斗与成本观察。发布前必须满足：

| 指标 | 通过条件 |
|---|---:|
| 测试用户正确理解订阅、自动续期和算力额度 | ≥ 90% |
| 成功取得权益后完成首个真实任务 | ≥ 85% |
| 并发预占导致用户或全局额度超支 | 0 |
| 失败、超时或重试造成重复扣减 | 0 |
| 供应商账单与内部成本账本差异 | ≤ 3% |
| P75 用户月模型成本 / 订阅净收入 | ≤ 25% |
| 设计压力场景中的模型成本 / 订阅净收入 | ≤ 30%，否则必须限额或调价 |

若质量门槛下仍无法控制尾部成本，优先调整模型路由、上下文、额度和订阅价格；不能靠隐藏用量、任务执行后突然截断或把虚拟公司破产包装为真实 AI 限额。

## 5. R3：App Store 与首发地区合规

### 5.1 已核实边界

Apple 当前规则要求：

- 有账号或后台能力时提供可用审核账号或完整演示模式；
- 向第三方 AI 共享个人数据前，明确说明共享对象并取得明确许可；
- 使用或展示第三方服务必须得到其条款允许；
- 解锁 App 内数字功能通常必须使用 App 内购买；外部购买链接只在特定地区和授权条件下允许；
- 中国大陆发行可能需要有效 ICP/APP 备案信息，相关元数据必须匹配。

中国大陆面向公众提供生成内容的服务还需要由专项法律意见判断《生成式人工智能服务管理暂行办法》、算法/生成式 AI 备案、内容治理、投诉处理、个人信息和跨境数据要求是否适用。**平台托管 AI 会强化平台对内容、数据、供应商和持续服务的直接责任，不能只依赖模型供应商的合规结论。**

### 5.2 推荐发行策略

- **公开首发建议只发布 iOS**，因为用户当前明确目标是 App Store；
- Android 在阶段 0 保留 Release 构建、生命周期和 CI 烟雾验证，避免平台能力锁死，但不把双商店同时公开发布作为首版默认承诺；
- 中国大陆与非中国大陆作为两条独立发行轨道；如果大陆专项法律意见、备案主体或供应商可用性未通过，则首发地区暂不包含中国大陆，不阻塞其他已批准地区；
- 持续 AI 服务、订阅周期算力额度、存储、后台编排和经营功能统一通过 App Store IAP 自动续期订阅提供；
- 首版不提供 BYOK、外部付款、网页购买引导或外部购买价差提示；虚拟公司资金不能映射为订阅权益或 AI 算力额度；
- 审核说明清楚解释订阅包含的持续服务、额度规则、模型由平台路由、发送前有供应商级同意，以及模型成本由平台承担；
- 审核环境提供沙盒 IAP、专用审核账号和真实可重复任务，不能只提供静态演示或要求审核员采购第三方服务。

### 5.3 合规门禁

在接入生产供应商或提交商店前，必须完成：

1. 冻结首发国家/地区、运营主体、数据区域、语言和年龄分级；
2. 逐项数据流图：用户输入、文件、平台供应商密钥、订阅交易、任务结果、日志、分析、推送、备份分别到哪里；
3. 供应商条款复核：API 接入授权、保留/训练、地区、子处理者、删除和品牌使用；
4. App Store 3.1、5.1.2、5.2.2 清单和审核说明；
5. 中国大陆轨道单独取得书面法律意见并完成适用备案，不以内部工程判断代替；
6. 隐私政策、App Privacy、App 内同意界面和真实数据流四者逐字段一致；
7. 完成购买、恢复购买、退款 / 撤销、宽限期、过期、账号删除、撤回第三方 AI 同意和导出的端到端演练。

同一审核原因连续被拒两次后，不继续盲目改文案重复提交；升级到 App Review 沟通 / 申诉，重新评估订阅商品、额度表达、审核演示和地区范围。

## 6. R4：平台 AI 供应商密钥安全

### 6.1 为什么平台仍承担责任

平台为所有订阅用户调用供应商 API，生产 Key 一旦泄露会造成平台账单损失、用户数据暴露和供应商封禁，并可能让所有用户同时失去 AI 能力。其爆炸半径高于单一用户凭据，必须按生产支付凭据级别保护。

### 6.2 解决方案

- 使用云 KMS / Secret Manager；数据库只保存凭据引用、版本和状态，不保存可直接调用的完整 Key；
- 按 Local / Development / Staging / Production、供应商和数据区域隔离凭据与服务身份，非生产环境不得使用生产 Key；
- 只有隔离的 AI Worker 身份可按执行尝试解密；API、Temporal、所有客户端、客服和分析系统均无解密权；
- 明文只在 Worker 内存中短暂存在，不进入文件、队列、Workflow History、错误、日志、Tracing、崩溃报告或客服界面；
- Key 写入后不可通过管理后台读取，只能验证、轮换、停用或删除；
- 所有使用记录执行 ID、Worker 身份、供应商配置 ID、环境、区域、时间和结果，但不记录密钥；
- 生产只允许已验证的 HTTPS 供应商域名，不向普通用户开放自定义 Base URL；
- 每个供应商配置具有独立日 / 月上限、速率限制和紧急停止开关，轮换或停用不依赖发布新 App；
- 建立密钥泄露响应：立即停止相关供应商路线、切换已批准备用、撤销 / 轮换供应商 Key、调查范围、通知受影响方并复盘。

### 6.3 安全发布门禁

- 用“假密钥金丝雀”扫描日志、追踪、错误、Temporal History、数据库普通字段、分析和崩溃报告，发现数必须为 0；
- IAM 负向测试证明非 AI Worker 无法解密；
- 轮换、停用、供应商撤销和 KMS 主密钥换代演练全部通过；
- 完成威胁建模、依赖扫描、SAST/DAST 和独立渗透测试；
- 所有严重 / 高危问题清零，中危必须有书面接受和截止日期；
- 发生疑似泄露时能在 30 分钟内停止相关供应商继续产生费用、保护用户数据并定位受影响范围。

## 7. R5：多 AI 供应商兼容

### 7.1 关键结论

“OpenAI-Compatible”不能作为全部多供应商能力的事实标准。OpenAI Responses、Anthropic Messages 和 Gemini `generateContent` 的端点、输入结构、流式事件、工具调用、结构化输出、用量、停止原因、安全拦截和文件能力都不同；同为兼容接口的厂商也常只兼容基础聊天。

### 7.2 首版协议策略

能力层预留以下四种 Adapter 类型，但首版生产路由只启用一个主供应商和一个已验证备用供应商，不以“支持厂商数量”作为发布目标：

1. `openai_responses` 原生 Adapter；
2. `anthropic_messages` 原生 Adapter；
3. `google_gemini` 原生 Adapter；
4. `openai_compatible` 最小协议 Adapter，只允许平台验证过的固定域名与模型组合。

若首发地区不支持上述供应商，保持同样的协议分层，用当地目标供应商替换；不能只改品牌名而跳过契约、价格、质量和数据政策测试。普通用户不创建连接、不配置 Base URL，也不能强制使用未验证模型。

统一能力不是简单的 `chat()`，而是：

```text
capability_profile
text_input
image_input
structured_output
tool_calling
streaming
usage_reporting
cancellation
data_region
retention_class
```

业务层只提出能力要求。Adapter 返回统一事件、标准错误和原始供应商诊断引用；不支持某能力时在任务开始前阻止，而不是执行到一半再失败。

### 7.3 契约门禁

每个已验证“供应商 + 协议版本 + 模型”组合必须通过：

- 平台认证、错误凭据、供应商额度不足、限流、超时、取消和重试；
- 普通响应、流式结束、异常中断和重复事件；
- JSON Schema/结构化输出和修复失败；
- 工具调用参数、并行工具、工具错误和回传；
- 图片/文件能力声明与实际调用；
- Token/用量字段缺失或变化；
- 数据同意、平台路线停用和故障切换不越权；
- 至少 60 条协议契约用例和对应任务评测切片。

发布标准：契约测试 100% 通过；故障注入恢复率 ≥ 99%；供应商、模型、价格或 Adapter 升级后自动重跑。未验证组合不得进入普通用户生产路由。

### 7.4 当前 DeepSeek APP PoC 的边界

当前体验验证可以先只实现 `deepseek_v4` Adapter，以降低首个垂直切片的变量数量；这不等于生产版放弃多供应商或备用路线。PoC 固定官方域名，启动时验证 `/models`，使用当前官方文本模型 ID，不依赖临近下线的旧别名。开发者 Key 只存在本地 / 测试服务端 Secret，严禁打包进任何客户端、仓库和日志。

PoC 只使用合成或非敏感数据，并完成流式、取消、JSON、Tool Calls、用量、错误映射和断点恢复。DeepSeek 官方条款与隐私政策所涉及的数据处理区域、下游用户告知和同意责任属于生产发布门禁；在首发地区、法律依据和数据路径未冻结前，不能把“接口已经跑通”宣传为“可向全球用户提供”。生产开放前仍必须完成一个备用供应商或明确的不可用降级路径、真实账单对账与地区合规审查。

## 8. R6：模型路由的质量、价格与延迟

### 8.1 产品承诺必须分层

平台可以保证任务系统、文件解析、引用结构、确定性计算和人格不改写事实，但不能假设最便宜的模型适合所有任务，也不能让高阶模型处理每个秘书摘要。模型价格、上下文、输出稳定性和限流会变化，因此质量承诺必须绑定到经过验证的“任务类型 + 路由策略 + 模型 / 价格版本”，而不是供应商品牌。

后台模型组合使用三种状态：

- **已验证**：指定模型版本通过该任务类型的完整评测；
- **能力受限**：可完成基础文字，但缺少视觉、工具、结构化输出或上下文能力；
- **未验证**：只能在隔离评测环境运行，不能进入普通用户生产路由。

### 8.2 解决方案与门禁

- 保留 `WorkProduct` 与 `PersonaReport` 两层，员工口吻不重写正式事实；
- 文件和表格先由确定性解析器处理，数值和公式由程序复核；
- 发布评测集不少于 450 条，三类任务各不少于 150 条；
- 每个已验证模型组合达到需求文档中的覆盖、引用、数值、人格漂移和工具越权阈值；
- 路由器先按任务能力、安全、地区和用户同意过滤，再在质量门槛内按预算和延迟选择轻量 / 均衡 / 高阶模型；
- 模型不满足任务能力或成本上限时在提交前清晰阻塞，不能静默降质或执行后截断；
- 用户可选择“快速 / 均衡 / 深度”等产品模式，但不能指定未验证模型或绕过额度；
- 供应商模型、价格、Prompt、工具、检索、路由阈值或输出 Schema 变更都重跑对应切片和成本压力测试。

这使“平台质量承诺”和“成本承诺”都绑定到可复现、可回滚的版本组合。

## 9. R7：状态组合爆炸

### 9.1 风险表现

一个任务可能同时处于：等待文件、AI 执行、员工加班、电话召回、员工请假、用户取消、模型限流、App 离线、场景表现落后、结算重试和工资日。若把这些状态直接写进页面、角色动效或手势条件分支，会出现重复结算、同人多位置、假进度和无法恢复。

### 9.2 解决方案

- Task、AI Attempt、Dispatch、Employee、Attendance、Ledger 和 Visual Projection 分别有独立状态机；
- PostgreSQL 保存业务事实，Temporal 保存可重放流程，React Native 客户端只消费带版本的视觉投影，不反向决定业务状态；
- 客户端投影按 `projection_version` 幂等应用，场景汇报、标签、气泡和角色姿态都可由服务端快照重建；
- 每个状态机只有一个写入方，跨域变化通过事务 Outbox；
- 结算、发薪、购买、加班和召回使用业务唯一键，不依赖“Workflow 只跑一次”；
- P0 HR 使用确定性规则，随机事件只影响表现和有限数值，不改变核心 AI 能力；
- 为核心不变量编写属性测试/模型测试，而不只覆盖几个顺利用例。

### 9.3 门禁

必须覆盖至少以下交叉故障：

- 任务提交重试 × 员工非工作时间；
- AI 完成 × App 被杀 × 推送重复；
- 用户取消 × 供应商晚到结果；
- 员工离职/请假 × 任务执行中；
- 结算提交 × 数据库故障 × Workflow 重放；
- 时区修改 × 发薪日/加班跨日；
- 客户端视觉投影事件丢失/乱序 × 服务端快照恢复；
- 前后台切换 × 场景聚焦/展开标签 × 新旧快照竞争；
- 账号删除 × 仍在运行的任务、文件和记忆。

所有核心不变量属性测试通过，连续故障注入不得产生重复收入、重复工资、余额不可重建、跨租户数据或 AI 完成前展示最终结果。

## 10. R8：React Native 轻量互动办公室、视觉一致性与内容产能

### 10.1 默认技术路线与场景结构

轻量互动办公室默认使用 React Native `View`、`Image`、`Pressable` 搭建可访问、可测试的界面结构，使用 Reanimated 驱动位移、缩放、透明度和状态过渡，使用 Gesture Handler 处理点击、拖拽、缩放与房间聚焦。只有角色数量、同屏动效或绘制开销经 Release 真机数据证明默认角色层不达标时，才把 **actors layer** 条件式迁移到 React Native Skia；场景 UI、业务交互、可访问语义和正式结果页仍留在 React Native。

场景采用可独立替换和测量的分层结构：

1. **背景层**：办公室整体背景、日夜色调与固定环境；
2. **房间层**：3—4 个房间容器、遮罩、聚焦区域和房间级装饰；
3. **家具/前景层**：可遮挡元素与少量状态变化资产；
4. **角色层**：角色、核心姿态和预设路径上的移动表现；
5. **信息层**：气泡、标签、任务状态与可访问点击区；
6. **交互层**：全景/房间聚焦切换、角色选择、返回与结果入口。

每个房间和角色使用稳定的**语义槽位**，例如 `room.finance.desk.primary`、`room.research.report`、`actor.accountant.home`；业务投影只引用槽位和状态，不写屏幕绝对坐标。角色移动仅使用产品预先验收的**预设路径**和停靠点，不引入自由寻路。不同 iPhone 画布通过布局约束、裁切安全区和槽位映射适配，不分别维护互相漂移的场景逻辑。

### 10.2 技术性能 Gate

技术性能与美术产能是两个独立 Gate；任一未通过都不能用另一项的结果替代。技术 Gate 必须使用 **Release 构建真机**，测试固定场景为：4 个房间、8 名员工、3 个并行动画、1 个气泡、2 个展开标签，连续运行 20 分钟，并覆盖全景、房间聚焦、角色选择、结果页往返和前后台切换。

通过条件：

- 交互与场景运动接近稳定 60 FPS；
- P95 UI 帧时间 ≤ 20 ms；
- >100 ms 的非加载帧 ≤ 1 次/分钟；
- 场景增量内存目标 ≤ 120 MB，并记录进入前基线、稳定峰值和退出后回收；
- 连续前后台切换 50 次，无崩溃、黑屏、角色重复、标签错位、聚焦错误或业务状态错乱；
- `output.ready`、场景汇报、原生结果页和返回快照恢复链路可重复，表现延迟不能提前暴露尚未完成的业务结果；
- 基准设备严重热状态、系统内存压力和弱网恢复均有记录，结果可复现。

默认 RN 角色层未达标时，先做两轮有数据归因的定向优化，包括减少重渲染、图片尺寸与缓存治理、动效并发控制、视口裁切和投影更新合并。两轮后仍不达标，才把角色层迁移到 Skia actors layer，并使用 Skia Atlas 等批量绘制能力复测；**不回到游戏引擎**。迁移不能改变语义槽位、预设路径、业务投影协议或结果页边界。

### 10.3 黄金视觉 Gate

批量资产生产前，必须完成一套可在真机审查的黄金视觉样板，范围固定为：

- 1 个竖屏办公室；
- 3—4 个房间；
- 4 个正式角色；
- 每个角色覆盖 6 个核心姿态；
- 日间与夜间两套完整观感；
- 全景和房间聚焦两种视图；
- 三种 iPhone 画布，至少覆盖窄屏、主流和大屏尺寸。

黄金样板必须由**产品负责人明确确认并留存签字/记录**，确认项包括品牌气质、信息层级、角色辨识度、房间可读性、日夜一致性、三种画布裁切和交互反馈。未通过或仅口头“基本可以”时，禁止批量制作角色、房间、姿态、装饰和昼夜变体。

### 10.4 资产一致性与内容产能 Gate

- 所有场景和角色资产遵守统一的透视、比例、色板、边线、光影方向、命名、画布和导出规则；
- 字体、插画、图标、纹理、动效及参考素材必须有明确商业授权、适用地区和可留档凭证；授权不明的资产不得进入原型之外的任何发布候选；
- AI 只用于概念探索和受控变体，不作为未经审校的最终资产流水线；
- 禁止把多批独立生成、透视/比例/色板/边线不一致的资产直接拼接进同一办公室；
- 每项资产必须经过视觉规范检查、三种画布检查、日夜检查、真机性能检查和产品验收后，才计为“有效资产”；
- 以“有效资产/人周”测量内容吞吐，按黄金样板后的真实数据估算剩余工作，并固定保留 30% 返工余量；
- 若吞吐不能覆盖发布日期，先减少外观变体、装饰、非核心姿态和房间内可选内容，不降低授权、视觉一致性和验收标准。

技术性能 Gate 和黄金视觉/内容产能 Gate 均通过后，才能冻结场景布局、同屏容量、资源格式、语义槽位、预设路径和资产吞吐基线。

## 11. R9：AI 与平台总成本及商业可持续

### 11.1 成本事实

平台订阅收入需要同时承担：

- 主、备 AI 供应商的输入、输出、缓存、工具和失败重试费用；
- KMS/Secret Manager；
- API、AI Worker 和 Temporal；
- PostgreSQL、Redis、对象存储和 CDN；
- 文件解析、OCR、搜索和文档生成工具；
- APNs/FCM、WebSocket、日志、追踪和崩溃平台；
- 备份、跨区、客服、安全和 App Store 运营。

如果按 App Store 标价、平均聊天轮数或营销口径“无限使用”定价，重文件、长上下文和高阶模型用户会快速造成负毛利；Apple 实际结算、退款和税费也会让可用于成本的净收入低于标价。

### 11.2 解决方案与门禁

- 分开记录 `ProviderUsageLedgerEntry`、`UserAllowanceLedgerEntry` 与 `platform_cogs`；供应商真实成本、用户额度和虚拟公司资金不能混入同一账本；
- 每个任务记录模型价格版本、输入 / 输出、缓存、工具、重试、CPU 时间、工作流时间、存储增长、出站流量、OCR / 搜索调用和日志量；
- 基于 App Store 实际净收入和内测真实任务混合冻结价格与额度，不用理论平均值定价；
- 订阅产品控制 AI 算力额度、后台编排并发、存储、保留、文件解析、搜索工具和经营功能；达到限额时清楚说明真实原因，不伪装成员工拒绝或公司资金不足；
- 建立免费 / 试用防滥用、单任务、单用户、供应商和平台全局上限，并提供实时紧急停用和受控降级；
- 每日对供应商账单与内部模型成本账本对账，每月同时复盘模型成本、平台成本、Apple 净结算、退款率和毛利；
- P0 使用托管服务和模块化单体，不自建 Kubernetes、多云数据库或工作流集群。

发布门槛：在 P50、P75、P95 三档真实用户负载和价格压力场景下完成单位经济测算；P75 用户模型成本不高于订阅净收入 25%，设计压力场景不高于 30%，供应商账单对账差异不高于 3%；模型成本加平台可变成本后仍需达到发布前冻结的最低毛利。P95 必须有明确额度或降级，不允许无上限亏损。

## 12. R10：公司经营循环的产品假设

### 12.1 当前判断

“用户会优先在自己的公司提问”“员工表达更有趣”“任务会产生虚拟资产并推动招聘和扩建”是合理且有差异化的产品逻辑，因此不把一般意义上的聊天粘性列为第一风险。

真正需要证明的是：公司经营层是否在不拖慢真实任务的前提下，提高第二次及后续任务发生率。

### 12.2 实验方案

内测中比较两种真实可用路径：

- A：完整公司表达、角色汇报、收益和资产目标；
- B：保留同样 AI 能力和结果，仅弱化办公室演出与经营反馈。

建议冻结以下判断标准后再看数据：

- 首个有效任务完成率不能因公司流程下降超过 5 个百分点；
- A 组 7 日内第二个有效任务比例相对 B 组提升至少 15%；
- A 组查看结算卡、购买首项资产或扩建的比例达到预设门槛；
- 客户端收到 `output.ready` 后必须在 5 秒内完成交付演出并开放结果入口，不能把这段强制观看时间误当参与度；
- 用户能正确区分虚拟公司利润、App Store 订阅权益和 AI 算力额度，并理解模型真实账单由平台承担。

需要足够样本再判断差异；建议封闭测试不少于 400 名进入首任务的用户，并按订阅产品、任务类型和额度消耗分层分析。若经营层没有提升复用，先调整反馈节奏和资产目标，不删除“公司”定位；若反而显著降低任务完成率，则默认缩短演出、把经营反馈后置。

## 13. R11：文件、Prompt Injection 与数据隔离

### 13.1 解决方案

- P0 只允许 PDF、DOCX、XLSX、CSV、TXT、PNG、JPEG，普通单文件 25 MB；
- 文件直传隔离存储，扩展名、MIME、文件签名、解压大小、页数和单元格共同校验；
- 解析器运行在无凭据、无内网访问、有限 CPU/内存/时间的沙箱；
- 外部文件、网页和检索文本始终是“不可信数据”，其中的指令不能提高工具权限；
- 首版不执行文件宏、脚本、任意压缩包或真实外部写操作；
- 所有查询、对象键、向量索引、缓存和 WebSocket 订阅都强制携带 `company_id` 并做成员校验；
- 使用 PostgreSQL RLS 和跨租户负向测试形成第二道防线；
- 删除来源时同步删除原文件、派生物、索引和相关记忆。

### 13.2 门禁

- 跨租户访问成功数必须为 0；
- Prompt Injection 对抗集中的工具越权成功数必须为 0；
- ZIP bomb、超大表格、损坏文件、恶意图片、XXE 和解析器崩溃均被隔离；
- 账号/文件删除在规定时间内从在线系统和索引消失；
- 文件处理失败不能泄露内部路径、供应商密钥或其他租户元数据。

## 14. R12：持续运行与支持能力

### 14.1 风险

本产品在 App 关闭后仍承诺任务继续、结果推送、工资/时间事件和数据恢复，本质上是持续在线服务。没有值班、告警和事故处理能力时，文档中的 99.9%、RPO/RTO 和密钥安全只是纸面指标。

### 14.2 解决方案与门禁

- 生产初期优先使用托管 PostgreSQL、Temporal、KMS、对象存储、日志和崩溃平台；
- 每个 P0 告警有责任人、严重等级、排障手册和升级链；
- 把“任务执行成功率”“积压时间”“平台 Key 访问失败”“成本预占异常”“账单对账差异”“重复结算”“文件解析异常”“推送失败”“跨租户拒绝”作为业务告警；
- 每季度进行数据库恢复、密钥事件、供应商故障和区域不可用演练；
- 公开 SLO 只承诺团队能够值守和测量的范围，内测阶段不提前对外承诺 99.9%；
- 支持用户查看任务执行记录、订阅权益、额度变化和恢复动作，降低人工客服对敏感原文的访问需求。

发布门槛：连续 14 天预发布 soak，核心任务成功/清晰可恢复率 ≥ 99%，账本对账差异为 0，备份恢复和回滚演练通过，P0 值班覆盖已实际排班。

## 15. 五级风险消减门禁

### Gate -1：立项与发行边界

必须冻结：

- 首发地区与是否包含中国大陆；
- iOS 首发、Android 技术验证/后续公开的范围；
- 首版主供应商、备用供应商、协议和模型档位；
- 平台订阅产品、IAP、算力额度和成本目标；
- 平台 Key 托管、第三方数据同意和供应商账单责任；
- 团队、预算、目标日期和值班能力。

出口：R1、R2、R3 有明确负责人和测试安排；订阅价格 / 额度、主备供应商和成本基数已冻结；大陆发行若未获批准则从首发地区移除。

### Gate 0：订阅、安全与成本原型

- APP 体验 PoC 可先用 DeepSeek 单供应商验证一条合成 / 非敏感文本任务；该例外只用于内部开发，不授予公开发布资格；
- StoreKit 2 购买 / 恢复、`appAccountToken`、App Store Server Notifications V2 和服务端权益周期跑通；
- 一个主供应商和一个备用供应商的真实调用、契约、故障切换和第三方数据同意跑通；
- 平台 Key 存储、访问、轮换、停用、紧急熔断和假密钥扫描通过；
- 原子成本预占、实际结算、双账本、价格版本和供应商账单对账通过；
- 购买页与额度理解可用性测试通过；
- 数据流图、隐私同意和供应商条款清单完成。

出口：证明用户能订阅、平台能安全调用且不会并发超支，错误能恢复并可对账。

### Gate 1：轻量互动办公室与黄金视觉 PoC

- 使用 React Native `View`/`Image`/`Pressable`、Reanimated 和 Gesture Handler 完成 1 个竖屏办公室，默认不启用 Skia 角色层；
- 完成 3—4 个房间、4 个正式角色、6 个核心姿态、日夜、全景/房间聚焦和三种 iPhone 画布的黄金视觉样板；
- 黄金样板由产品负责人明确签字；签字前禁止批量资产生产；
- 先完成“一名核心员工 + 一名秘书 + 一个 DeepSeek 文本任务”的体验链路，再按 4 房间、8 员工、3 并行动画、1 气泡、2 展开标签做 Release 真机 20 分钟压力验收；
- 验证 `output.ready → 场景汇报 → 原生结果页 → 返回后快照恢复`，并确认正式结果只来自业务事实，场景投影丢失或乱序后可由快照恢复；
- 技术性能 Gate 与黄金视觉/内容产能 Gate 分别验收，任何一项不得代替另一项；
- 默认 RN 两轮定向优化仍不达标时，只把角色层迁移到条件式 Skia actors layer 并复测，不改变业务和场景协议。

出口：冻结场景布局、同屏容量、语义槽位、预设路径、资源格式、三种画布规则、设备范围和有效资产/人周吞吐，并保留 30% 返工余量。

### Gate 2：最薄真实垂直切片

用真实服务完成：

```text
账号 → IAP 权益 → 成本预占 → 真实任务 → 云端后台 → 员工演出
→ 员工走到汇报点 → React Native 正式结果 → 追问/修订 → 一次幂等收入
→ AI 实际成本结算 → 一次发薪 → 购买一项资产
→ App 被杀后恢复 → 供应商账单对账
```

出口：同一链路连续运行、故障注入和数据恢复通过后，才扩三类任务和完整浅 HR。

### Gate 3：封闭内测

- 三类任务与 450 条评测通过；
- 不少于 100 名用户验证订阅、额度理解与首任务漏斗；
- 足够样本验证公司经营实验；
- 完成模型成本分布、订阅净收入、供应商账单对账、单位经济、数据删除、备份恢复、渗透测试和 14 天 soak；
- 所有严重/高危缺陷清零。

出口：指标达到事先冻结阈值，而不是测试后修改阈值解释成功。

### Gate 4：App Store 发布

- 沙盒 IAP、审核账号、购买与恢复、审核说明、隐私标签、额度规则和供应商同意流程可完整演示；
- 生产后台在审核期间可用；
- 首发地区要求和备案材料全部匹配；
- 采用分阶段发布，异常时可停止放量并回滚服务端、模型配置和场景资源清单。

## 16. 不允许绕过的停止条件

以下任一发生时必须停在当前阶段：

1. 团队/预算/日期无法覆盖冻结范围和 30% 余量；
2. 订阅权益用户的首个真实任务完成率在两轮迭代后仍低于 85%，或额度理解正确率低于 90%；
3. 未取得首发地区所需法律意见、备案或供应商授权；
4. 平台供应商 Key 出现在任何日志、追踪、历史、客户端或普通数据库字段；
5. 严重/高危安全问题未关闭；
6. 已验证 Adapter 契约不能 100% 通过；
7. 450 条评测未达到质量门槛；
8. 默认 RN 轻量场景两轮定向优化仍不通过 Release 真机性能门槛，并且团队拒绝把角色层迁移到 Skia actors layer；
9. 黄金视觉未由产品负责人明确签字，却开始批量生产角色、房间、姿态、装饰或昼夜变体；
10. 任一字体、插画、图标、纹理、动效或参考素材商业授权不明，或无法提供适用范围凭证；
11. 多批独立生成且视觉规范不一致的资产被直接拼接，或有效资产/人周与 30% 返工余量无法支撑冻结范围；
12. 故障注入能产生重复成本扣减、额度超支、重复虚拟结算、余额不可重建或跨租户访问；
13. 供应商账单与内部成本账本差异超过 3% 且无法解释；
14. P75 模型成本超过订阅净收入 25%，或压力场景超过 30%，且无法通过限额、路由或调价解决；
15. 模型成本与平台可变成本合计在合理限额下仍不能由平台净收入覆盖；
16. 审核拒绝原因未理解却继续用相同方案重复提交；
17. 没有实际值班和恢复演练却准备公开承诺后台可靠性；
18. DeepSeek Key、供应商响应原文或用户任务内容进入客户端包、仓库、普通日志或分析事件；
19. 生产仍使用已下线模型别名，或未完成目标地区的数据处理、告知同意与供应商条款审查。

停止不是项目失败，而是阻止昂贵问题进入更晚阶段。

## 17. 仍需产品负责人冻结的六项决定

这些决定不能靠继续技术搜索替代：

1. **首发地区**：是否包含中国大陆；
2. **公开平台**：是否接受“iOS 首发，Android 先做技术验证”；
3. **供应商与模型路线**：全球轨道和大陆轨道各自的一主一备、轻量 / 均衡 / 高阶模型及价格版本；
4. **订阅商品**：免费 / 试用范围、各订阅档 IAP 价格、周期算力额度、并发和存储上限；
5. **成本与毛利底线**：除模型成本 25% / 30% 约束外，平台总可变成本允许占净收入的比例；
6. **资源约束**：个人开发者可实际承担或采购的团队角色、预算、目标日期和值班方案。

在这六项冻结前，可以做 Gate 0/1 的可逆原型，不应承诺公开发布日期。

## 19. 最终判断

本项目最合理的落地路线不是缩成一个普通聊天 App，也不是立即并行制作全部 HR、经济、场景内容和多模型系统，而是：

1. 先冻结首发地区、iOS 范围、订阅商品、净收入成本基数、团队和发布资源；
2. 先证明 IAP 权益、平台密钥、原子成本预占、双账本和供应商账单对账；
3. 用一主一备的真实生产候选证明多供应商 Adapter、模型分层和授权内故障切换；
4. **先通过 React Native 轻量场景结构 Gate 和黄金视觉 Gate**：冻结分层、语义槽位、预设路径、布局、容量、资源格式、三种画布规则和有效资产吞吐；
5. 用 Release 真机数据验证默认 RN 实现；仅当角色层两轮优化后仍不达标时迁移到 Skia actors layer，不改变业务协议；
6. 用真实纵向链路证明任务、场景汇报、正式结果、AI 成本结算、虚拟经济、HR 和后台恢复；
7. 最后扩展完整但浅的 P0 HR 与经营闭环，并通过内测证明公司表达确实提高复用。

本计划不存在游戏引擎或旧式降级选择。轻量场景的技术性能与黄金视觉/内容产能必须分别通过，且黄金视觉未签字前不得批量生产。只要严格执行 Gate -1 到 Gate 4、不绕过停止条件，并为六项产品决定提供明确答案，当前重大风险都有超过 90% 把握被发现、隔离或解决。若跳过这些门禁，现有文档再详细也不能提供同等把握。
