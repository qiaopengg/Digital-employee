## 1. 结论

办公室继续使用当前固定前方高位俯视镜头，禁止通过改镜头、临时缩放人物或增加遮挡掩盖动作问题。

首版人物主链路选择：

```text
React Native 0.86 New Architecture
+ Reanimated 4.5.3 / Worklets 0.11.1
+ React Native Skia Atlas
+ 多方向序列帧动作资产
+ 确定性运动学控制器
+ 固定导航图与时空预约
+ 家具 / 人物 / 道具接触点约束
```

这里追求的是“在固定 2D 镜头下没有可见物理穿帮”，不是刚体、布料、碰撞反弹或真人级动作捕捉。员工不能被家具弹开、被碰撞旋转或因惯性滑行，因此不引入 Matter.js、Box2D 或独立游戏引擎。人物位置、朝向、速度、脚步、交接物和座椅接触由可验证的运动学规则决定。

Rive 具备骨骼、IK、约束、状态机和 React Native 运行时，但不作为首版人物主链路。原因是当前角色风格和固定俯视动作需要逐帧可控的轮廓、遮挡、脚步相位与接触点；新的 React Native Rive 运行时还需要 Nitro Modules，且运行时主要通过状态机推进动画，不如序列帧路线容易进行逐帧距离匹配和黄金截图回归。Rive 可用于后续局部 UI 动效，或在独立人物 PoC 证明质量、资源和真机并发均更优后重新评审。

对该方案“能否在当前 React Native 0.86 项目中实现”的工程信心为约 95%。剩余风险集中在动作资产质量、纹理驻留内存和低端真机帧时间，必须通过第 10 章 PoC Gate 消除，不能用文档判断代替真机证据。

## 2. 当前实现审计

现有实现已经有可复用基础，不需要推倒全部场景：

- `officePhysicsModel.ts` 已有标准化坐标、静态碰撞体、线段与扩张矩形碰撞检测、人物半径和并行路径重叠检查；
- `officeSeatModel.ts` 已有工位座位、站立点、朝向和前景遮挡概念；
- `officeMotionAnimation.ts` 已有按路径分段移动和按路程分配各段时长的基础；
- `OfficeSeatForegroundLayer.tsx` 已证明背景可拆出家具前景遮挡层；
- `SceneSnapshot`、语义槽位和服务端视觉意图不需要因渲染升级而改变。

当前问题不是“缺少更多 if”，而是表现层的模型不够完整：

1. 行走只有两帧，步态由时间循环驱动，脚步和实际路程没有一一对应；
2. 起身通过 `scaleY` 与平移模拟，直接造成离开工位时人物变小；
3. 姿态切换使用透明度过渡，形成双人重影；
4. 转向只是切换图片方向，没有转身片段、减速和脚步收势；
5. 座椅接触只依赖一个角色图片锚点和背景裁片，没有骨盆、脚、手与椅背的共同约束；
6. 交接双方走到通道中央，缺少“到对方工位对接”的工作因果；
7. 文件交接没有唯一所有者事件，视觉上可能同时拿着、突然消失或瞬间出现；
8. 气泡在普通工作、移动、回座等状态常驻，与最新产品规则冲突；
9. 路径只校验静态家具和两条等分路径，不支持多人物的时间段占用、让行和目的地预约；
10. 当前交接行为集中在一个 Hook 中，继续叠加喝水、如厕、返岗和多员工逻辑会变成不可维护的条件堆。

所以应复用场景模型与碰撞基础，替换角色渲染、动作驱动和行为编排，不能继续向当前两帧方案追加补丁。

## 3. 候选路线评估

| 候选 | 优点 | 主要问题 | 结论 |
|---|---|---|---|
| RN `Image` + 两帧切换 | 无新依赖、已有实现 | 无法满足转身、脚底锁定、接触和无重影要求 | 淘汰 |
| RN 多 `Image` 序列帧 | 逻辑直观 | 多人物多帧会产生较多原生视图和透明图合成 | 仅作静态回退 |
| Skia `Atlas` + Reanimated | 官方支持 RN 0.86 所在版本范围；适合批量精灵；帧、位置和图层可控 | 增加原生依赖和包体；需要动作资产流水线 | 首选 |
| Rive 骨骼 / IK | 骨骼、IK、约束、状态机、资源复用强 | 新 RN 运行时增加 Nitro 依赖；逐帧距离匹配与外部家具接触需要额外验证；可能出现纸片关节感 | 后续候选 |
| Matter.js / Box2D | 完整刚体碰撞 | 会引入不需要的质量、摩擦、反弹、旋转和求解不稳定；不能自动解决动作质量 | 淘汰 |
| RVO 动态避障 | 适合大量持续移动人群 | 本项目最多 3 人同时移动，且通道固定；局部避让可能抖动、偏离通道或卡窄路 | 首版不采用 |
| 固定导航图 + 时空预约 | 确定、可复现、易测试，适合少量演员和固定办公室 | 需要预先定义通道节点、让行点和预约规则 | 首选 |

选择 Skia 不是因为当前 RN 一定跑不动，而是人物已经确定需要多方向、多片段的序列帧。官方 `Atlas` 就是为大量相似纹理、精灵和动画变换设计，并能与 Reanimated worklet 直接结合；继续用大量独立 `Image` 只是把确定要发生的迁移推迟。

## 4. 客户端职责分层

```text
OfficeWorkspaceScreen
├── OfficeShell
│   ├── CompactHUD
│   ├── OfficeViewport
│   │   ├── RN Background / Rear Furniture
│   │   ├── Skia ActorsAndPropsCanvas
│   │   └── RN Front Masks
│   ├── NativeInteractionOverlay
│   │   ├── ActorHitTargets
│   │   ├── StatusIcons
│   │   └── WorkHandoffBubble
│   ├── NewTaskButton
│   └── SecretaryDock
└── Native Sheet / Page Host
```

- Skia 只绘制人物、随身道具和必要动态遮挡；
- 背景、家具静态层、状态图标、气泡、点击热区、秘书总控、Sheet 和无障碍节点继续使用 RN；
- 画布节点不承担业务状态或无障碍，透明原生热区与场景等价列表提供完整操作；
- `SceneSnapshot` 与 `visual_intent` 仍是事实输入，动作完成只清理本地队列；
- 镜头矩阵固定，角色世界比例在同一办公室模板内固定，所有动作片段使用同一根节点尺度。

推荐的新职责边界：

| 模块 | 职责 |
|---|---|
| `OfficeBehaviorPlanner` | 把视觉意图或短时生活规则编排为动作序列 |
| `OfficeRoutePlanner` | 在固定导航图上选择合法路径 |
| `OfficeRouteReservation` | 预约节点、边和交互槽位的时间段，处理让行 |
| `OfficeKinematicController` | 每帧更新根位置、朝向、速度、路程和动作相位 |
| `OfficeInteractionController` | 对接、交接、入座、沙发、饮水等原子互动 |
| `OfficeActorCanvas` | 依据动作帧、根位置、深度和遮挡绘制 |
| `NativeInteractionOverlay` | 点击、状态图标、气泡、文字与无障碍 |

现有 `useHandoffBehavior.ts` 不继续扩张为全办公室行为引擎；它在迁移完成后只保留编排入口，或被上述小模块替代。

## 5. 人物运动模型

### 5.1 权威运动状态

每名可见员工只维护一个本地运动状态：

```ts
type ActorMotionState = {
  actorId: string;
  rootPosition: { x: number; y: number };
  heading: 'n' | 'ne' | 'e' | 'se' | 's' | 'sw' | 'w' | 'nw';
  speed: number;
  distanceTravelled: number;
  clipId: string;
  clipPhase: number;
  routeReservationId?: string;
  interactionReservationId?: string;
  heldPropId?: string;
};
```

React 状态只保存低频语义变化；`rootPosition`、`speed`、`distanceTravelled` 和 `clipPhase` 在 Reanimated UI worklet 中更新。每帧回调只读取紧凑的数值状态，不跨线程创建对象或触发 React 重渲染。

### 5.2 行走与脚底锁定

走路不能再按固定时间切两帧。循环步态以实际累计路程驱动：

```text
clipPhase = (distanceTravelled / clip.strideLength) mod 1
```

每个行走片段在 manifest 中记录左右脚接触区间、单周期真实位移和允许速度区间。角色根节点沿路径移动，当前帧由 `clipPhase` 采样；同一路程始终对应同一脚步相位。变速通过改变根节点推进速度完成，不通过随意加快或减慢帧循环造成滑步。

开始、停止和转弯使用独立片段：

```text
站立
→ 起步预备
→ 第一步落地
→ 循环行走
→ 提前按制动距离进入收步
→ 双脚稳定
→ 转身或交互
```

路径方向和当前朝向夹角超过动作允许范围时，根节点停止平移，先播放转身片段。任意移动帧都必须面向速度方向；不允许沿返回路径倒播朝向，也不允许先平移后补转身。

### 5.3 路径与速度

办公室使用固定导航图，不进行像素级自由寻路：

- 节点：工位出口、主通道拐点、门口、让行点、茶水点、如厕出入口、会议入口和老板汇报点；
- 边：可走通道，声明长度、宽度、方向限制、遮挡区间和容量；
- 静态碰撞：复用现有“线段与扩张家具矩形”检查，人物半径通过 Minkowski 扩张进入碰撞判定；
- 动态碰撞：路径规划后预约节点和边的时间区间；后来的员工选择等待点、延后出发或另一条合法路径；
- 同时最多 3 人移动，使用优先级与预约表即可，不接入连续 RVO；
- 员工到达路径末端后必须进入停止和目标对齐，不能直接把坐标吸附到终点。

预约优先级：等待老板 / 汇报交付 > 真实任务交接 > 会议 > 返岗 > 饮水 / 如厕 / 伸展。低优先级行为可以被取消并重新规划，高优先级交接不能穿过已占用员工。

### 5.4 深度与遮挡

- 人物深度以脚底根节点所在的路径区段决定，不以图片顶部或数组顺序决定；
- 家具拆成后层、接触层和前景遮挡层；
- 走到桌后、椅前、门框后时使用 manifest 声明的遮挡区间；
- 点击热区随根节点移动，但不能改变绘制深度；
- 调试模式显示脚底根节点、碰撞圆、路径、预约区间、家具 collider、接触点和当前深度。

## 6. 家具与互动约束

### 6.1 固定语义槽位

每件可互动家具必须声明：

- `footprint`：不可穿越占地；
- `approachSockets`：人物允许接近的位置和朝向；
- `contactSockets`：骨盆、左右脚、左右手、道具和视线目标；
- `frontMasks`：会遮挡人物的前景图层；
- `allowedClips`：该家具允许播放的动作；
- `capacity`：同时占用人数；
- `releaseRule`：何时释放预约。

固定办公室模板按目标容量预放工位。没有分配员工的工位保留电脑、桌椅和桌面物件，但没有人物、状态图标或使用动画。

### 6.2 工位坐下与起身

工位至少具有 `seatSocket`、`standSocket`、`egressSocket`、`computerLookTarget` 和椅子前景遮挡。坐下人物使用专门的工位坐姿序列帧，骨盆根点与 `seatSocket` 对齐，脚和手的接触误差在资源验收中检查；不再把站立人物缩短后叠在椅子上。

```text
返回工位出口
→ 面向椅子
→ 对齐 standSocket
→ 播放坐下片段
→ 在片段接触事件锁定 seatSocket
→ 切换工位工作循环
```

起身反向使用独立片段，但不是倒放坐下动画。椅子保持静止，人物尺度保持不变；必要时由椅子前景层遮挡腿部和座面接触。

### 6.3 沙发与休息区

沙发每个座位有独立 `seatSocket`、朝向、骨盆高度、脚落点和前景遮挡。只允许播放针对该沙发透视制作的坐下、坐姿和起身片段。员工不能使用工位坐姿图片叠到沙发，也不能沿沙发靠背方向坐反。

### 6.4 工作交接

默认工作交接发生在接收者工位旁，而不是双方都走到通道中央：

```text
派件员工预约路径和 receiver.visitorSocket
→ 起身、转向、沿通道行走
→ 在 visitorSocket 停步并面向接收者
→ 接收者转椅 / 起身到 receiver.receiveSocket
→ 显示工作交接气泡
→ 双方播放伸手片段
→ prop_release 关键帧把文件所有者从派件人切为空中交接点
→ prop_acquire 关键帧把文件所有者切给接收者
→ 收手、确认、气泡消失
→ 派件人转身后返回，接收者入座工作
```

道具任一帧必须只有一个权威视觉所有者：员工、交接点、桌面或收纳点。所有者变化由动作关键帧事件触发，不由 setTimeout 猜测。对话气泡只在上述工作对接、复核、交接或汇报窗口出现；走路、等待、返岗、工作、饮水和如厕只显示状态图标。

## 7. 工作时间行为调度

工作时间的基础状态是 `at_desk`，不是 `idle_roaming`。员工没有任务动画时也坐在自己的工位阅读、整理或等待，不长时间站立和随机游走。

短时活动必须同时满足：

- 当前没有不可中断的工作对接；
- 目的地槽位可预约；
- 存在完整往返路径；
- 具备开始时间、最晚返回时间和冷却时间；
- 同一区域的短时活动人数不超过模板规则；
- 新任务、会议、汇报或老板事项触发时可取消、返岗或直接改道；
- App 恢复时根据当前时间和服务端快照收束，不补播完整生活动画。

首版短时活动只包括饮水、如厕、短暂同事交流和伸展。上厕所只走到洗手间出入口并离开可视场景，不在办公室内演出隐私动作。普通短时活动不产生对话气泡，不伪造 AI 进度，也不影响底层任务执行。

具体持续时间、冷却和同时人数由版本化配置决定，需在黄金样板中调节；技术实现不得把魔法数字散落在组件内。

## 8. 状态 UI 与导航

- 收起导航固定为独立“新建任务”主按钮和屏幕边缘“秘书总控”；
- 秘书总控收起时只显示秘书图标和待办数，展开后显示任务、公司、汇报、我的及管理入口；
- 员工常态状态使用轮廓不同的图标和语义色，不常驻文字标签；
- 点击、长按、VoiceOver 和场景等价列表显示员工姓名、岗位和准确文字状态；
- 颜色不是唯一信号，每种状态必须拥有稳定图标形状和无障碍名称；
- 气泡只有 `work_event_id` 存在且事件类型为对接、交接、复核或汇报时才能出现；
- AI 仍在处理但没有交接事件时，最终负责人保持工作姿态与状态图标，详细进度在点击后的任务卡 / Sheet 中展示。

## 9. 动作资源协议

序列帧不只是图片列表。每个片段必须提供机器可校验的 manifest：

```json
{
  "clipId": "walk_with_folder_east_v1",
  "actorArchetype": "office_adult_a",
  "direction": "e",
  "fps": 16,
  "loop": true,
  "frameRects": [],
  "rootAnchor": {"x": 0.5, "y": 0.92},
  "worldScale": 1,
  "strideLength": 0.048,
  "footContacts": {
    "left": [[0.0, 0.2]],
    "right": [[0.5, 0.7]]
  },
  "events": [],
  "compatibleFrom": ["walk_start_east_v1"],
  "compatibleTo": ["walk_stop_east_v1", "turn_e_to_ne_v1"]
}
```

所有人物动作共享同一透明画布、根锚点约定和世界尺度。允许不同动作拥有不同透明边界，但根节点不能因此跳动。关键片段至少包含：工位坐姿、工位工作、起身、八方向起步 / 行走 / 收步、必要转身、交接伸手 / 接收 / 收手、工位入座、沙发入座 / 坐姿 / 起身、饮水和汇报。

纹理按当前可见员工和动作族按需加载；压缩文件大小和 GPU 解码后内存分别统计。首包只放黄金办公室核心角色和静态回退，其他角色资源按版本下载、校验、缓存和回滚。

## 10. 技术与视觉 Gate

引入新依赖前先获得确认；确认后只做一个办公室、两名员工、一次完整工位交接的垂直 PoC。PoC 通过才扩展人物和生活行为。

### 10.1 自动验收

| 项目 | 通过条件 |
|---|---|
| 世界比例 | 同一员工所有片段和路径中的运行时 scale 变化为 0 |
| 静态家具穿越 | 全部预设路径与扩张 collider 相交数为 0 |
| 人物相撞 | 三人压力夹具的节点 / 边时间预约冲突为 0 |
| 倒退行走 | 移动方向与朝向不一致帧数为 0；转身期间根节点不平移 |
| 道具所有权 | 任一帧恰有一个视觉所有者；交接事件顺序固定 |
| 气泡真实性 | 无 `work_event_id` 时气泡数量为 0 |
| 空工位 | 未分配工位有人物、状态或使用动画的数量为 0 |
| 恢复 | 中断任意动作后按快照进入合法姿态，不重播已完成交接 |

### 10.2 视觉验收

- 以正常速度和 0.25 倍速观看完整“工作坐姿 → 起身 → 转身 → 行走 → 停步 → 交接 → 转身 → 返回 → 入座”；
- 人物脚底在接触帧无可见滑移，身体无双影、瞬切、缩放和倒退；
- 工位与沙发坐姿在固定镜头下看起来是坐在家具中，不是覆盖在家具上；
- 文件从一只手移动到另一只手时没有复制、消失或跳跃；
- 所有动作在当前冻结镜头下验收，不提供“换角度后正常”的豁免。

### 10.3 真机性能验收

使用 Release 包运行 4 个房间、8 名可见员工、3 人同时移动、1 个工作交接气泡和 2 个展开任务标签，持续 20 分钟；复用《技术设计.md》第 9 章的帧时间、内存、前后台和结果可见门槛。Skia 官方说明其当前版本线要求 React Native ≥0.79、React ≥19；项目 RN 0.86 / React 19 满足前置条件。官方列出的包体增量约为 iOS 6 MB、Android 4 MB，必须纳入发布预算。

## 11. 实施顺序

1. 冻结当前办公室镜头矩阵、设计画布、人物世界尺寸和两套工位 / 沙发接触点；
2. 引入并锁定经注册表依赖约束确认兼容 RN 0.86 的 Reanimated 4.5.3、Worklets 0.11.1 与 Skia 2.9.1，跑双平台空画布构建；
3. 建立动作 manifest、Skia Atlas 绘制和原生热区叠加，只渲染一个静态员工；
4. 实现距离驱动步态、起步 / 收步 / 转向，替换当前两帧和 `scaleY` 起身；
5. 复用并扩展现有 collider，增加固定导航图、时空预约和调试图层；
6. 实现一个“到同事工位交接”的原子互动，接入道具关键帧所有权；
7. 实现工位和沙发接触约束，移除背景裁片式临时遮挡；
8. 将气泡限制到工作事件，状态改为图标与颜色，接入秘书总控；
9. 增加空工位和工作时间基础行为，再增加饮水、如厕、短交流和伸展；
10. 运行自动、视觉和真机 Gate；任何一项失败都不扩展角色资产。

## 12. 调研依据

- [Reanimated 与 React Native 兼容矩阵](https://docs.swmansion.com/react-native-reanimated/docs/guides/compatibility/)：4.4–4.6 版本线明确覆盖 RN 0.86，Reanimated 4 只支持 New Architecture；
- [Reanimated `useFrameCallback`](https://docs.swmansion.com/react-native-reanimated/docs/advanced/useFrameCallback/)：逐帧 worklet 在 UI 线程执行；
- [React Native Skia 安装与版本要求](https://shopify.github.io/react-native-skia/docs/getting-started/installation/)：当前版本线要求 RN ≥0.79、React ≥19，并给出双平台包体增量；
- [React Native Skia Atlas](https://shopify.github.io/react-native-skia/docs/shapes/atlas/)：面向批量精灵 / tile，并可用 worklet 低成本动画变换；
- [React Native Skia 与 Reanimated 动画集成](https://shopify.github.io/react-native-skia/docs/animations/animations/)：Shared Value 可直接驱动 Skia 属性并在 UI 线程执行；
- [Rive React Native 新运行时迁移说明](https://rive.app/docs/runtimes/react-native/migration-guide)：新运行时基于 Nitro Modules，并支持文件缓存和多实例；
- [Rive IK Constraint](https://rive.app/docs/editor/constraints/ik-constraint)：IK 可用于手部目标和脚底约束；
- [Rive Transform Constraint](https://rive.app/docs/editor/constraints/transform-constraint)：可复制位置、旋转和缩放并控制约束强度；
- [Unreal Engine Root Motion](https://dev.epicgames.com/documentation/unreal-engine/root-motion-in-unreal-engine)：动作位移与角色根节点需保持一致，避免网格与碰撞根分离和回弹；
- [Unreal Engine Locomotion](https://dev.epicgames.com/documentation/en-us/unreal-engine/locomotion-in-unreal-engine)：距离匹配用目标距离驱动动画播放，可减少速度变化造成的滑步；
- [Unreal Engine 修复脚滑](https://dev.epicgames.com/documentation/unreal-engine/fix-foot-sliding-with-ik-retargeter-in-unreal-engine?lang=en-US)：使用脚部速度曲线识别并锁定脚掌接触区间；
- [AAAI Cooperative Pathfinding](https://ojs.aaai.org/index.php/AIIDE/article/view/18726)：在时空中规划并用 reservation table 避免多角色路径冲突；
- [Godot NavigationAgent 说明](https://docs.godotengine.org/en/4.3/tutorials/navigation/navigation_using_navigationagents.html)：避障与寻路是分离问题，RVO 可能在窄路停顿或离开导航边界，适合作为持续移动对象的补充而非固定通道的默认方案；
- [Matter.js 官方文档](https://brm.io/matter-js/docs/index.html)：其核心是质量、摩擦、反弹、刚体、约束和求解器，这些能力并不能解决本项目的动作接触与序列帧质量，反而扩大状态空间。
